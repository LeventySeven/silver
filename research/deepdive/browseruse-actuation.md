# browser-use — actuation lens (registry, multi_act, watchdogs, sensitive-data)

Sources read in full: `tools/registry/service.py` (611 lines), `agent/service.py:2718-2853` (`multi_act`), `browser/watchdogs/{permissions,crash,downloads,popups,security}_watchdog.py`, and Silver's `src/security/registry.ts`, `src/actuation/{actions,pagechange}.ts`, `src/security/redact.ts`. Cross-checked against the seed digest at `research/topfive/top5-browseruse.md` (items 2-5), which this expands with line-level mechanism detail and a re-argued gap analysis, especially on sensitive-data (I disagree with the seed's severity framing there — see below).

## 1. Action registry: decorator → normalized-signature → auto-Pydantic-model → dispatch

`Registry.action(description, param_model=None, domains=None, terminates_sequence=False)` (`service.py:291`) is a class decorator factory. Decoration triggers `_normalize_action_function_signature` (`service.py:75-272`), which does three things in one pass over `inspect.signature(func).parameters`:

1. **Special-param carve-out.** `_get_special_param_types` (`service.py:57-73`) is a hardcoded dict of ~9 names (`browser_session`, `page_url`, `cdp_client`, `page_extraction_llm`, `available_file_paths`, `has_sensitive_data`, `file_system`, `extraction_schema`, `context`) that the framework injects at call time rather than the LLM supplying. Any function parameter whose name matches one of these is pulled out of the action's public schema and type-checked against the expected type (`service.py:113-144`, including Optional-unwrapping and `issubclass` compatibility) — a mismatch raises at decoration time, not at call time, so a plugin author who annotates `browser_session: str` by mistake gets an immediate `ValueError`, not a runtime surprise mid-task.
2. **Auto-model synthesis.** If no `param_model` is supplied, `create_model(f'{func.__name__}_Params', __base__=ActionModel, **params_dict)` (`service.py:159`) builds a Pydantic model directly from the remaining (non-special) parameters' type annotations and defaults — so `async def click(index: int, browser_session: BrowserSession)` becomes a fully schema-validated LLM tool with zero boilerplate.
3. **Kwargs-only normalization.** The decorator returns a `normalized_wrapper(*args, params=None, **kwargs)` (`service.py:169-258`) that rejects positional args, reconstructs the original call by walking `parameters` in declaration order and slotting in either `params_dict[name]` (action arg) or `kwargs[name]` (special arg) — this is what lets `execute_action` (`service.py:331-419`) call every action through one uniform `action.function(params=validated_params, **special_context)` regardless of how the original function was written (Type 1: first arg is the whole param model; Type 2: individual typed args).

`create_action_model` (`service.py:517-603`) then builds the LLM-facing tool schema: one Pydantic model per action containing a single field (`{action_name: (param_model, Field(description=...))}`), unioned via `RootModel` when there's more than one action, filtered by `domains` against the current `page_url` (`registry._match_domains`) so e.g. a Gmail-only action never appears in the tool list on a non-Gmail page. This domain-scoped tool-set filtering is itself notable — it shrinks the LLM's action surface per-page rather than presenting a static global menu.

**Mechanism vs Silver's registry.ts.** Silver's `buildRegistry(flags)` (`security/registry.ts:118-125`) is a pure function from two booleans (`enableActions`, `readOnly`) to a `Set<string>` union of two hardcoded arrays (`READ_ONLY_VERBS`, 34 entries; `ACTOR_VERBS`, 24 entries). There is no decorator, no dynamic registration, no per-function schema synthesis — verb dispatch in `actions.ts` is a closed `ActVerb` union type resolved through a switch, and parameter validation is whatever TypeScript's type system + manual checks in each handler enforce, not an auto-generated schema. This is architecturally the opposite of browser-use's registry, and correctly so per the file's own header comment: "a disabled verb is literally NOT in the dispatchable set, so no prompt... can make the CLI dispatch it." Adopting browser-use's dynamic `action()` decorator would reintroduce exactly the attack surface (an LLM-influenced or plugin-influenced action set) that Silver's registry is designed to statically foreclose. **Recommendation: do not adopt the registration mechanism.** The one transferable idea — auto-deriving a param schema from a function's type signature instead of hand-writing it — has no real payoff in TS since Zod schemas are already declarative and about as much code as a decorated Python function; skip this too. Priority: N/A (correctly rejected).

## 2. `multi_act`: two-layer abort with partial-result preservation

`agent/service.py:2720-2838`. Before dispatching a batch of actions produced by one LLM turn, the loop captures `pre_action_url = await self.browser_session.get_current_page_url()` and `pre_action_focus = self.browser_session.agent_focus_target_id` (line 2769-2770) immediately before each individual action call (not once for the whole batch — this matters because it means the guard is evaluated fresh every iteration, catching drift introduced by the *previous* action in the same batch). After the action returns, two independent checks can abort the remaining queue:

- **Static:** `registered_action.terminates_sequence` (line 2806) — a boolean declared at registration time on actions known to always change page context (navigate, search, go_back, switch_tab). This catches page-changing actions even when the runtime check would race (e.g., navigation still in flight).
- **Runtime:** `post_action_url != pre_action_url or post_action_focus != pre_action_focus` (line 2816) — catches page changes the static flag didn't anticipate (a click that triggers an unexpected redirect, a form submit).

Either guard breaks the loop with a logged reason and skip-count (`f'skipping {total_actions - i - 1} remaining action(s)'`). On exception, the except block (line 2820-2836) explicitly re-raises `InterruptedError` and connection-like errors (so `_handle_step_error` can still do reconnect/shutdown logic upstream) but for ordinary action exceptions appends `ActionResult(error=...)` to `results` and returns the partial list — the caller (the agent's step loop) sees exactly which of N queued actions succeeded before the failure, rather than an opaque batch failure. There's also a narrow rule at line 2750-2755: `done` may only be the sole action in a batch, and if it appears mid-batch execution simply stops there.

**Silver comparison — functionally equivalent via a different mechanism, one narrower gap.** Silver's `pagechange.ts` computes `settleAndFingerprint` (`url|focusedBackendNodeId|domNodeCount`, `pagechange.ts:69-83`) after a bounded settle (domcontentloaded + a race against a 1.2s network-idle budget, line 71-75) and returns it as a `page_changed`/`stale_refs` **flag** on the single-command envelope, per the file's own design note (lines 4-9): the CLI must never auto-embed a fresh snapshot or auto-decide anything — that's the host LLM's job, consistent with Silver's one-verb-per-invocation model (there is no LLM-planned "batch of actions" the CLI itself executes; the host issues one CLI command, reads the flag, decides the next command). The equivalent safety property — never act on a stale ref — is enforced earlier and more strongly than browser-use's approach: `groundRef` (`perception/refmap.ts`, called at `actuation/actions.ts:135`) checks a **generation counter** and fails the *specific* command with a typed error before any Playwright call touches the page, rather than waiting for a post-hoc URL/focus diff. This is arguably tighter than browser-use's fingerprint (which can miss changes that alter neither URL nor focus nor node count, e.g. an SPA re-rendering the same DOM node count with different content).

The real gap is narrow and already correctly scoped by the seed digest: `cli.ts`'s `batch` verb chains multiple CLI commands in one process but (per file inspection) has no mid-batch abort-on-page-change beyond `--bail` on hard error — it does not stop a queued batch early just because an earlier command in it changed the page, the way `multi_act`'s two-layer guard does. **Recommendation: adopt selectively** — in `batch`, after each sub-command, check the fingerprint delta and stop (skip-with-reason, mirroring browser-use's logged skip-count) rather than running all remaining sub-commands against a page that already changed out from under them, even though the *next single command* would independently fail-safe via `groundRef`. Priority: **P2** (safety is not compromised today — `groundRef` already prevents silent stale-ref corruption — but batch UX/efficiency improves: failing fast saves wasted Playwright round-trips on refs the CLI can already tell are dead before dispatch).

## 3. Watchdogs: narrow, portable wins inside a large architecture Silver should not import wholesale

The 13-file `browser/watchdogs/` directory (9,305 total lines) is a `bubus` event-bus fan-out: each watchdog declares `LISTENS_TO`/`EMITS` class-level lists of event types and implements `on_<EventName>` handlers (e.g. `PermissionsWatchdog.LISTENS_TO = [BrowserConnectedEvent]`, `permissions_watchdog.py:18`, granting CDP permissions via `Browser.grantPermissions` on connect, line 37-39 — 43 lines total, one of the simplest watchdogs). `CrashWatchdog` (336 lines) runs a background `_monitoring_loop` (line 226) polling `_check_network_timeouts` and `_check_browser_health` and registers a raw CDP callback `on_target_crashed` (line 99) per attached target, emitting `BrowserErrorEvent` on detection. `DownloadsWatchdog` is the largest single watchdog (1,483 lines) — it tracks in-flight downloads via both Playwright's page-level download event and a raw CDP `Browser.downloadWillBegin`/`downloadProgress` listener pair, maintains an initial-files snapshot to diagnose downloads that land without a CDP event firing (`_initial_downloads_snapshot`, line 136), and exposes direct callback registration (`register_download_callbacks`, line 146) that bypasses the event bus entirely for latency-sensitive "wait until this specific download completes" call sites.

**Silver gap: complete absence, not partial.** `find ... -iname "*download*" -o -iname "*dialog*" -o -iname "*permission*"` against `silver/src` returns **zero files** — there is no download watcher, no permission-grant step, no crash-monitoring loop, and no dialog/popup handler module anywhere in Silver's source tree. `cli.ts` has a single regex-based crash classifier (`/crash/i.test(msg)`, `cli.ts:116-133`) that maps a raw Playwright error message to the `page_crash` error code post-hoc — it detects a crash only after a command already failed against one, it does not proactively monitor target health. There is no code path for "the page opened a download" or "the browser is requesting camera/geolocation permission" — commands that hit either scenario presumably hang or fail unhandled today (not verified against a live download-triggering page in this pass — flagging as a testable gap, not a confirmed live failure).

**Recommendation: adopt the narrow mechanisms, not the event-bus architecture.** Silver's CLI is single-process, one-verb-per-invocation, with no long-lived agent loop to subscribe watchdogs to — importing `bubus` and 13 files of event-driven services is disproportionate machinery for that shape. But three specific, portable mechanisms are worth lifting directly into Silver's connection/session layer:
- **Permission auto-grant on connect** (mirror `permissions_watchdog.py:37`): a one-line `Browser.grantPermissions` CDP call in Silver's session-connect path, gated by a CLI flag, would prevent commands from silently stalling on a native permission prompt Playwright/CDP can't auto-dismiss.
- **Download detection + path capture**: even a minimal version (a `page.on('download')` handler wired once per session that resolves with the saved path, without the full CDP-callback/duplicate-detection machinery) would close the current zero-coverage gap for any task that clicks a download link.
- **Proactive crash detection**: registering a `Target.attachedToTarget`/`crashed` CDP callback at session-connect time (rather than only classifying the error message after a command already threw) would let Silver return a clean `page_crash` immediately instead of whatever raw timeout/connection error currently surfaces first.

Priority: **P1** for download+permission handling (concrete, currently-unhandled functional gap on tasks a browser agent routinely needs — file-download workflows, cookie-consent-style permission prompts); **P3** for proactive crash detection (the reactive classifier already produces the right error code, just later than ideal).

## 4. Sensitive-data: `<secret>` indirection — re-argued severity vs the seed digest

`tools/registry/service.py:427-514` (`_replace_sensitive_data`) plus the call site in `execute_action` (line 354-365, 372, 377-379). Mechanism: the agent is constructed with `sensitive_data: dict[str, str] | dict[str, dict[str, str]]`. In the new (domain-scoped) format, each top-level key is a domain glob pattern (e.g. `"https://*.bank.com"`) mapping to a `{label: value}` dict; the LLM is shown only the labels (via the system prompt), never the values. When the LLM wants to type a credential, it emits `<secret>label</secret>` as the literal text in an action's string parameter (e.g. the `input`/`type` action's `text` field). Only at execution time, after Pydantic validation of the action params but immediately before the Playwright call, does `_replace_sensitive_data` regex-match `<secret>(.*?)</secret>` (line 442) and substitute the real value — and only for secrets whose domain pattern matches the *current* page URL via `match_url_with_domain_pattern` (line 458), so a `bank.com`-scoped secret is structurally inapplicable if a prompt-injected page instructs the agent to type it into `evil.com`. A secondary feature: labels suffixed `bu_2fa_code` are treated as TOTP seeds and expanded via `pyotp.TOTP(...).now()` (line 475, 491) rather than a static value — first-class 2FA support. The special-param plumbing further restricts blast radius: `has_sensitive_data` and `sensitive_data` are only injected into the `input` action's special-context dict (line 372, 377-379), not into every action's context, so an action author can't accidentally leak the dict through an unrelated action.

Silver's `src/security/redact.ts` (54 lines) is the mirror-image control: it operates on the **read** path only. `redactValue(role, name, rawValue, isPassword)` (line 44-54) is the choke point every serializer-emitted node value passes through, replacing password-shaped or card-shaped values with `[redacted]` before they ever reach a snapshot the host LLM sees. Confirmed by grep: no `<secret>`, `vault`, or `fromEnv` handling anywhere in `cli.ts` or `actuation/actions.ts`; `fill`/`type` (`actions.ts:31-57`, confirmed by direct read) take the value as a literal string argument with no indirection layer.

**Where I diverge from the seed digest's framing:** the seed calls this "the largest security-relevant gap found" and frames it as tension with Silver's keyless threat model. I'd sharpen that: it is not in tension with the *keyless* claim (Silver still never calls a model, regardless of whether secrets flow through it) — it's in tension with a **different, unstated guarantee** that browser-use's design happens to provide "for free" as a side effect of being agent-orchestrated: the secret value never has to enter *any* LLM's context window, including the host LLM driving Silver. In Silver's architecture, the host LLM is the one constructing the `silver fill --ref e7 --value <literal>` command line, which means the literal secret must already be resolvable in whatever produced that command — the host LLM's own context (if it read the secret from a doc/prompt) or a value it echoes from an env var it was told to read. Silver being "keyless" (no model API key) says nothing about whether *credentials the task requires* pass through the host LLM's tokens; browser-use's mechanism specifically prevents that regardless of which LLM is driving it.

**Recommendation: adopt — this is a real, currently-open gap, not a hypothetical.** Concretely: add a `--secret name=value` CLI flag or `SILVER_SECRET_<NAME>` env-var registration mechanism (resolved by the CLI process itself, never round-tripped through the host LLM), have `fill`/`type` in `actuation/actions.ts` resolve a `<secret>name</secret>` pattern in the incoming `value` at the same choke point where `redactValue` already sits on the read side (symmetric read/write chokepoints is a clean design), and domain-scope registered secrets against the current page URL exactly as browser-use does (`match_url_with_domain_pattern` is a ~20-line glob matcher, trivially portable). TOTP/2FA support is a nice-to-have, not required for parity. Priority: **P1** — this is the one item in this lens where Silver's actual security posture (not just feature parity) is measurably weaker than browser-use's for any task involving real credentials, and the fix is small (one new module + two call-site edits) relative to its payoff.
