# Deep Dive: browser-use perception layer vs. Silver `walk.ts`

Lens: DOM serializer + interactive-element heuristic cascade + selector-map + viewport/visibility filtering.

Source read in full: `browser_use/dom/serializer/serializer.py` (1290 lines), `browser_use/dom/serializer/clickable_elements.py` (247 lines), `browser_use/dom/serializer/paint_order.py` (213 lines), `browser_use/dom/service.py` (lines 46-901, the tree-construction + visibility pipeline), `browser_use/dom/views.py` (`EnhancedDOMTreeNode`, `is_actually_scrollable`, `should_show_scroll_info`, lines 380-720). Cross-checked against `research/sources/r2-browseruse-dom.md` and `research/topfive/top5-browseruse.md`. Silver comparison: `silver/src/perception/walk.ts` (561 lines, read in full) and `silver/src/perception/serialize.ts`/`roles.ts` (skimmed for gap confirmation).

## What browser-use does, and how (mechanism)

**1. Three-source join, not a single tree walk.** `dom/service.py` fires three CDP calls in parallel per snapshot — `DOMSnapshot.captureSnapshot` (layout/paint-order/computed-style/rects), `DOM.getDocument({depth:-1, pierce:true})` (full DOM incl. shadow roots), `Accessibility.getFullAXTree` per frame (merged across frames, `_get_ax_tree_for_all_frames`, service.py:347-391) — then joins all three by `backendNodeId` into one `EnhancedDOMTreeNode` (`_construct_enhanced_node`, service.py:722-1018). A 10s timeout with one retry at 2s (`asyncio.wait`, service.py:564-622) raises rather than silently returning partial state. The selector-map key the LLM outputs IS the CDP `backendNodeId` (`serializer.py:713`) — no synthetic ID space, no ref-mint/resolve step, O(1) dict lookup.

**2. Five-stage serialization pipeline** (`serializer.py:100-148`, `serialize_accessible_elements`):
- Stage 1 `_create_simplified_tree` (435-540): recurses DOM+shadow-roots+iframe content documents, drops `DISABLED_ELEMENTS` (`style/script/head/meta/link/title`) and 15 decorative `SVG_ELEMENTS` (path/rect/g/circle/…) outright (lines 18-38), force-includes hidden validation elements (`aria-*`/`pseudo*` attrs) and hidden file inputs (opacity:0 custom-styled pickers, common Bootstrap pattern).
- Stage 2 `PaintOrderRemover.calculate_paint_order` (`paint_order.py`): groups nodes by CDP `paint_order`, walks highest-to-lowest, and for each node checks whether its bounding rect is *already fully covered* by higher-paint-order rects already added to a `RectUnionPure` (a disjoint-rectangle union with rect-splitting, capped at 5000 rects to bound worst-case exponential fragmentation, `paint_order.py:52`). Transparent/near-transparent (opacity<0.8) elements don't count as occluders. This removes elements truly hidden under modals/overlays that CSS-visibility alone can't detect.
- Stage 3 `_optimize_tree`: prunes structurally-empty wrapper nodes bottom-up, keeping file inputs as an explicit exception.
- Stage 4 `_apply_bounding_box_filtering` (729-838): a **containment-collapse** pass — if a node is one of eight `PROPAGATING_ELEMENTS` patterns (`<a>`, `<button>`, `div/span[role=button|combobox]`, `input[role=combobox]`), its bounds propagate to *all* descendants, and any descendant ≥99%-contained gets `excluded_by_parent=True` (rendered as flattened text, not a separate ref) — UNLESS it's a form element, has its own onclick, meaningful aria-label, interactive role, or is itself a propagator (5 exception rules, lines 802-838). This is what stops `<a><span>Home</span><svg>...</svg></a>` from producing 3 refs for what's semantically one link.
- Stage 5 `_assign_interactive_indices_and_mark_new_nodes` (617-727): the actual clickability decision (delegates to `ClickableElementDetector.is_interactive`, cached per-node), plus scrollable-container indexing (only if the container has *no* interactive descendants, unless it matches dropdown role/class heuristics — `listbox/menu/combobox/menubar/tree/grid` role, `select` tag, or `dropdown`/`select-menu` class substrings, lines 672-706) and new-node diffing against `previous_cached_state.selector_map` for change-highlighting (`*[123]` prefix).

**3. `ClickableElementDetector.is_interactive`** (`clickable_elements.py`): an ordered cascade, each layer a cheap short-circuit before the next: (a) `has_js_click_listener` — set upstream via a `getEventListeners()` devtools-only `Runtime.evaluate` call with `includeCommandLineAPI:True` batched across all elements (service.py:453-544, bails if >10k elements) — this is the only mechanism here that catches React/Vue/Angular synthetic-event handlers with zero DOM footprint; (b) large-iframe size check (>100×100px); (c) `label`/`span` wrapper detection via `has_form_control_descendant` (max depth 2, handles `label > span > input` Ant-Design-style compound widgets, explicitly skips `label[for=...]` to avoid double-activating the proxied input — a named bug fix comment references apartments.com); (d) search-icon heuristic — class/id/data-attr substring match against `{search, magnify, glass, lookup, find, query, search-icon, search-btn, searchbox}` (lines 76-103); (e) AX property scan (`disabled`/`hidden` short-circuit false, `focusable`/`editable`/`settable`/`checked`/`expanded`/`pressed`/`selected`/`required`/`autocomplete`/`keyshortcuts` short-circuit true); (f) native tag allowlist; (g) attribute/role allowlist (`onclick`, `tabindex`, ARIA interactive roles); (h) AX role allowlist; (i) icon-sized-element heuristic (10-50px bbox + any of `class/role/onclick/data-action/aria-label`); (j) final fallback: `cursor:pointer` computed style.

**4. Compound-component synthesis** (`_add_compound_components`, serializer.py:150-333): native controls that are *visually* one element but *semantically* several get virtual child descriptions injected into the attribute string — `<input type=range>` → synthesized `slider(min=,max=)`; `<input type=number>` → increment/decrement buttons + textbox with min/max; `<input type=color>` → hex textbox + picker button; `<input type=file>` → browse button + a textbox showing the **actual selected filename** (extracted from AX `valuetext`/`value`, cross-platform path-splitting on `\`/`/`, lines 234-256); `<select>` → dropdown toggle + up to first-4-option preview with a `format_hint` (numeric / country-state-codes / date-path / email) inferred by regex-sniffing the option values (`_extract_select_options`, 335-415); `<details>` → toggle + content-region; `<audio>`/`<video>` → play/mute/progress/volume/fullscreen controls. ISO date-format hints (`YYYY-MM-DD` etc.) are injected as an explicit `format=` attribute for native date/time inputs AND detected jQuery/Bootstrap/AngularJS-UI datepickers (class-name and `data-date-format`/`uib-datepicker-popup` sniffing, `_build_attributes_string`, 1102-1177) — specifically to stop LLMs guessing locale-formatted (MM/DD vs DD/MM) dates.

**5. Viewport/visibility filtering, not strict viewport-only.** `is_element_visible_according_to_all_parents` (service.py:241-345) checks CSS `display:none`/`visibility:hidden`/`opacity<=0` first, then a **configurable ±1000px threshold** (`viewport_threshold`, default 1000, settable to `None` to disable) beyond the actual viewport per nested-iframe frame chain, walking the html-frame ancestry in reverse and adjusting bounds by each iframe's own scroll offset (lines 290-343) — this correctly handles elements visible-but-scrolled inside a nested scrollable iframe, not just the top document. For elements that fall outside even that threshold *inside an iframe*, `hidden_elements_info` (service.py:100-182) collects up to 10 of the nearest hidden interactive elements with a computed `pages_down` scroll distance and surfaces them as an explicit hint line in the serialized output (`... (N more elements below - scroll to reveal): <button> "Submit" ~2.3 pages down`, serializer.py:1074-1080) — turning "invisible content the LLM doesn't know exists" into an explicit, actionable signal without spending tokens rendering the full hidden subtree.

## Why this beats naive AX-tree walks (and beats Silver here)

The paint-order occlusion pass and the bbox-containment collapse are the two pieces that most directly reduce **hallucinated affordances**: without them, a modal-covered background button is still "clickable" per AX role even though a real click would hit the overlay, and a styled `<a>` wrapping a `<span>` + decorative `<svg>` mints 2-3 refs for one link. Both are real failure modes in production sites (component libraries, modal-heavy SPAs) that a role-only walk cannot detect because ARIA roles carry no paint/z-order information. The compound-component synthesis and date-format hints are pure model-output-quality wins — they don't change what's clickable, they change whether the LLM picks the right *value format* on the first try, which is a measurable retry-count reducer for date pickers and range sliders specifically.

## The concrete gap vs. Silver's `walk.ts`

Silver's `walk.ts` is architecturally lighter: one `page.evaluate` cursor/hidden scan (`SCAN_JS`, walk.ts:498-555, checks `getComputedStyle().cursor==='pointer'`, `onclick`, `tabindex`, `contenteditable`, plus a hidden-radio/checkbox exception) tags matched elements, then `DOM.getDocument` + `Accessibility.getFullAXTree` per frame are joined by `backendNodeId` and DFS'd (walk.ts:256-343) with `refEligible = isInteractive || (isContent && name!=='') || cursorInteractive` (line 312). This is real convergent design — same backendNodeId-as-key idea, same cursor-interactive-cascade-for-AX-ignored-elements idea, same per-frame AX splicing idea (walk.ts:333-339 vs. browser-use's per-frame `getFullAXTree`) — but it is missing four mechanisms browser-use has and Silver doesn't:

1. **No paint-order / occlusion filtering at all.** Silver has no `DOMSnapshot.captureSnapshot` call and thus no `paint_order`/rects data source — `walk.ts` never requests `DOMSnapshot`, only `DOM.getDocument` + `Accessibility.getFullAXTree`. A button under an open modal or a covered-by-overlay element is indistinguishable from a genuinely clickable one to Silver's ref-eligibility check. **Priority: HIGH.** This is a correctness gap (wrong-affordance clicks), not just a quality-of-life one, and modal-heavy SPAs are common automation targets.
2. **No bounding-box containment collapse.** No `PROPAGATING_ELEMENTS` equivalent — a `<a><span>Text</span><svg/></a>` produces separate `refEligible` entries for the span (if it has meaningful AX content) and possibly the svg-wrapping structure, rather than one collapsed ref. Confirmed via `roles.ts`/`walk.ts` read: `refEligible` is computed purely per-node from role/name/cursorInteractive with no ancestor-containment check. **Priority: MEDIUM** — noisier snapshots (more refs than a human would count as "one button"), not a wrong-click risk since Silver's ref resolution targets `backendNodeId` directly rather than relying on bbox click-point math, but it inflates token cost and ref-count confusion.
3. **No viewport-threshold visibility model — Silver includes the whole page unconditionally.** `walk.ts`'s only spatial check is inside `SCAN_JS` (`rect.width===0||rect.height===0` skip, for the cursor-interactive scan only) — the main AX-driven walk has **no bounds check whatsoever**, so a `role`+`name`-bearing node anywhere on an infinitely-scrolling page becomes `refEligible` regardless of scroll position, with no `hidden_elements_info`-style "N more below, ~X pages down" hint to compensate. This is the inverse failure mode from browser-use: not under-inclusion (missing scrolled content) but **unbounded inclusion** (no per-viewport budget), meaning snapshot token cost scales with total page DOM size rather than viewport-relevant DOM size on long pages. **Priority: HIGH** for token-efficiency (the task brief's stated axis) — this is a bigger lever than the serializer's line-format compactness, since it's O(page) vs. O(viewport).
4. **No compound-component synthesis or date-format hints.** Already flagged in `top5-browseruse.md` #1; confirmed independently here. `<input type=date>`, `<input type=range>`, `<select>` all render as a bare `role`/`name`/`value` line in Silver with no min/max/option-preview/ISO-format signal. **Priority: MEDIUM** — model-output quality, not correctness; cheap to port since `walk.ts`'s `SnapNode.flags`/`attrs` already carries the raw DOM attributes needed (`min`, `max`, option children would need a small extra DOM read).

## Recommendation (keyless, adoptable)

All four are keyless — pure CDP/DOM mechanism, no model calls, consistent with Silver's architecture. Priority order: **(1) viewport-threshold visibility (configurable, default ~1000px like browser-use) + hidden-content hint line** — biggest token-efficiency lever, directly serves the task brief's snapshot-format-efficiency question, and is the one gap actively working *against* Silver's stated efficiency story right now; **(2) paint-order occlusion filtering** — requires adding a `DOMSnapshot.captureSnapshot` call (new CDP surface for Silver, needs `paint_order`+`bounds`+`computed-styles` fields) and a `RectUnionPure`-equivalent, moderate implementation cost, high correctness payoff; **(3) bbox-containment collapse** — same new bounds data unlocks this cheaply once (2) is done, small additional code; **(4) compound-component/date-hint synthesis** — lowest engineering cost of the four (no new CDP calls, just attribute synthesis in `serialize.ts` off existing `SnapNode.flags`/DOM attrs), do it opportunistically alongside any serialize.ts touch.
