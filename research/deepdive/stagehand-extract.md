## What Stagehand does, and how (mechanism)

Stagehand's `extract()` is not a single LLM call over raw HTML — it is a **schema-transform + snapshot-grounding pipeline** with two sequential LLM calls per invocation:

1. **Snapshot capture** (`captureHybridSnapshot`, `understudy/a11y/snapshot/capture.ts`): a 5-step multi-frame CDP pipeline — scoped fast-path for a single-selector target, one `DOM.getDocument` per CDP session (not per frame; sessions shared across same-origin frames), per-frame `Accessibility.getFullAXTree`, absolute cross-frame XPath prefix computation via `DOM.getFrameOwner`, and a final merge into `{combinedTree, combinedXpathMap, combinedUrlMap}`. Every element gets an `encodedId = "{frameOrdinal}-{backendNodeId}"` (`capture.ts:242-243,384,406`) — the frame-ordinal prefix specifically defeats CDP's `backendNodeId`-collides-across-iframes correctness bug.
2. **Pruning** (`a11yTree.ts:160-224,262-278`): drops `generic`/`none`/`inlinetextbox` roles with no name/children, collapses single-child structural wrappers, strips StaticText children whose text duplicates the parent's accessible name (kills "Sign in Sign in" duplication), replaces `generic`/`none` roles with the actual tag name, and folds `combobox+<select>` to role `select`.
3. **Schema transform** (`utils.ts:400-607`, `transformSchema`): recursively walks the caller's Zod schema (object/array/union/intersection/optional/nullable/pipe/effects, dual v3/v4 compat) and, for every leaf detected as `.url()` (via `checks.some(c => c.kind==="url" || c.format==="url" ...)`), swaps it for `makeIdStringSchema()` = `z.string().regex(/^\d+-\d+$/).describe("This field must be the element-ID in the form 'frameId-backendId'...")` (`utils.ts:672-686`). Path tracking uses a `*` sentinel for array indices so `list[].link` is found (`utils.ts:462-482`). Non-object schemas get wrapped in `{value: schema}` (`WRAP_KEY`) so a bare `z.string()` extract still has an addressable path.
4. **Inference call 1** (`inference.ts:34-124`): `buildExtractSystemPrompt` + `buildExtractUserPrompt(instruction, domElements, screenshotDataUrl?)`, `response_model:{schema, name:"Extraction"}`, `top_p:1, frequency_penalty:0, presence_penalty:0`. Anthropic path uses a `print_extracted_data` tool-call instead of native structured output.
5. **Inference call 2** (`inference.ts:53-64,154-199`): a SEPARATE "is this instruction actually satisfied" call against `metadataSchema = {progress: string, completed: boolean}` — decoupling format-compliance (call 1) from semantic completeness (call 2), used to drive chunked/paginated extraction loops. Both calls' token counts are summed into `{prompt_tokens, completion_tokens, reasoning_tokens, cached_input_tokens, inference_time_ms}` returned to the caller via `onMetrics()`.
6. **URL reinjection** (`utils.ts:616-661`, `injectUrls`): walks the *result object* along the same recorded paths and replaces every `ID_PATTERN`-matching value with `idToUrlMapping[id] ?? ""` — a live post-processing pass, not another LLM call.

The killer mechanism, stated once: **the LLM is never given the ability to type a URL for a URL-typed field.** It can only emit a regex-constrained ID that was enumerable from the snapshot it was just shown. This is reused verbatim as the `elementId` contract for `observe()` and `act()` (`inference.ts:277-313,425-461`) — one regex (`/^\d+-\d+$/`), one wire format (`[frameOrdinal-backendNodeId]`, shown literally in brackets in the tree text, e.g. `[0-18372]`), three independent schemas. `act`'s `action` field is `.nullable()` with explicit anti-fabrication language ("do NOT fabricate or guess an element, and never emit empty strings or placeholder values") repeated 3× across `prompt.ts`.

**Why this beats naive competitors** (browser-use, AgentQL free-text extraction, raw-HTML-to-LLM approaches): URL/link hallucination in extract is a top real-world failure mode when an LLM is asked to "read the href out of the text" — it invents plausible-looking but wrong paths/domains/query strings. Stagehand makes this *structurally* impossible rather than prompt-discouraged, and the same ID-space trick means act/observe selectors are equally hallucination-proof (the model can't invent a CSS selector because it never sees the concept — only a finite enumerable ID menu). The two-call extract further separates "did I extract in the right shape" from "does this satisfy the instruction," giving callers a clean stop signal for pagination without re-parsing free text.

**ActCache self-heal (`v3/cache/ActCache.ts`, read in full above):** cache key = `sha256({instruction, normalizeUrlForCacheKey(url), sortedVariableKeys})` (`:188-199`; URL normalization sorts query params so equivalent URLs collapse). `tryReplay` (`:71-146`) checks version===1, non-empty actions, exact variable-key-set match (sorted compare, `:362-377`), and — if variables are required — that all values are present in context *before* replay (`:110-124`), else it's a clean cache miss rather than a partial replay. `replayCachedActions` (`:201-285`) executes each cached `Action` via `waitForCachedSelector` (waits up to `domSettleTimeoutMs ?? 15000` for `state:"attached"`, logs-and-proceeds rather than hard-failing on timeout) then `handler.takeDeterministicAction` — **zero LLM calls on a hit**. After replay, `haveActionsChanged()` (`:287-325`) does a field-by-field diff (selector/description/method/arguments) between the cached actions and the just-executed live-resolved ones; if they differ, `refreshCacheEntry()` (`:327-360`) silently overwrites the `${cacheKey}.json` file with the healed version — cache is trusted for the fast path but self-repairs when the DOM has drifted instead of erroring or requiring manual invalidation. `CacheStorage` (`CacheStorage.ts:14+`) is a thin dual-backend abstraction: `fs.promises.readFile/writeFile` under a resolved dir (`mkdir -p`, pretty-printed JSON) or an in-memory `Map` (`createMemory`), both routed through `jsonClone()` to prevent aliasing bugs on returned objects; `enabled = !!dir || !!memoryStore`.

## Concrete gap vs Silver

Silver already **has and, in two places, hardens** the flagship id-grounding mechanism:

- `silver/src/extract/transform.ts` (`transformSchema`, `ID_PATTERN='^\\d+-\\d+$'`) is an explicit credited port of Stagehand's `transformSchema`/`makeIdStringSchema`, reworked for plain JSON Schema (no Zod — stays keyless). It additionally forces a `list[T]` container by default (`ensureContainer`, `transform.ts:107-118`) to defeat the "N-options-collapse-to-1" failure mode, which Stagehand does not do structurally (it relies on prompt language "YOU MUST EXTRACT ALL").
- `silver/src/extract/resolve.ts` (`resolveIds`) is the reinjection side, and it fixes Stagehand's own flagged anti-pattern: Stagehand's `injectUrls` falls back to `idToUrlMapping[id] ?? ""` on an unresolvable ID — a hallucinated ID silently becomes an invisible empty string. Silver's `resolveIds` instead nulls only the unresolved field and returns a named warning listing every offending ID (`resolve.ts:62-70`), and gates the whole resolution on `bundleGeneration === currentGeneration` (`resolve.ts:44-47`) — a staleness check Stagehand's `injectUrls` has no equivalent of at all.
- `silver/src/extract/prompts.ts` ports the verbatim `EXTRACT_SYSTEM_PROMPT`, credited and unedited, including the load-bearing "you MUST respond with ONLY the IDs of the link elements" clause.

**Real, unclosed gaps** (confirmed by reading both codebases fully — not just the top5 pass, which already flagged these; this pass verifies mechanism detail and priority):

1. **No action/extract cache at all — HIGH priority, cheap, keyless-compatible.** Silver has no `ActCache`/`CacheStorage` equivalent. `silver/src/task/store.ts` persists a run's plan/log/checkpoints (crash-recovery/replay-as-artifact — a different concern: "the script is the artifact"), and `silver/src/memory/store.ts` is grep-first episodic markdown with no structured selector state. Every repeat of the same instruction against the same normalized URL pays a full snapshot + host-reasoning round-trip, even for `task` re-runs or `subagent` loops hitting the same page shape repeatedly. **Adopt:** port `CacheStorage`'s two-backend design (fs-JSON-under-`mkdir -p`, or in-memory `Map`, both keyless — no model call in the cache path itself) and a `resolve`-cache keyed on `sha256({instruction, normalizedUrl, sortedUrlFieldPaths})` → last-good `{ref, backendNodeId, urlFieldPaths, valueMap-shape}` resolution hint. On hit, skip straight to `groundRef`/`toLocator` (Silver already owns the exact refmap/backendNodeId primitives Stagehand's cache stores) with **zero host round-trip**. Self-heal: diff the replayed resolution's live role/name against what's cached (Silver already computes accessible-name at snapshot time in `perception/accessible-name.ts`); on mismatch, silently rewrite the cache file rather than erroring — same shape as `haveActionsChanged`/`refreshCacheEntry`. This is Silver's single biggest latency lever from this source: since Silver never calls a model, a resolution cache converts "repeat this flow" from *snapshot + host-inference-round-trip* to *snapshot + cache-lookup*, which is a much larger relative win for Silver than for Stagehand (Stagehand's win is "skip the LLM call"; Silver's win is "skip the LLM call **and** possibly skip re-serializing/re-sending the tree to the host at all" if the cache is trusted enough to bypass the snapshot step for a known-stable target).

2. **No network-quiet wait before snapshot capture — MEDIUM priority.** Stagehand's `waitForDomNetworkQuiet()` (`actHandlerUtils.ts:528-684`) tracks in-flight `Network.requestWillBeSent` counts (minus finished/failed/cached/data-url), sweeps stalled requests >2s old, declares quiet 500ms after in-flight hits zero, hard-caps at `domSettleTimeoutMs` (default 5000ms) — explicitly built to replace Playwright's own `networkidle`, which Playwright deprecated as flaky. Silver's `silver/src/actuation/wait.ts` only exposes `networkidle` as a caller-opt-in spec; nothing in the snapshot/act path auto-waits for network quiescence, so a snapshot taken immediately after a click that kicks off async content can race a still-loading DOM. **Adopt:** port the in-flight-request-counter + stalled-sweep heuristic (CDP `Network.enable` + `requestWillBeSent`/`loadingFinished`/`loadingFailed` listeners) as an automatic pre-snapshot gate, not just a caller-opt-in wait spec.

3. **No DOM-depth retry ladder — LOW priority, cheap defensive win.** `DOM_DEPTH_ATTEMPTS=[-1,256,128,64,32,16,8,4,2,1]` steps down on CDP timeout/OOM for pathological deeply-nested pages. Silver's `walk.ts:173` calls `DOM.getDocument({depth:-1, pierce:true})` once with no fallback — a single unlimited-depth timeout fails the whole snapshot. **Adopt:** wrap the existing single call in a retry loop over a fixed depth-cap list, surfacing to the caller that the tree was depth-limited.

4. **Two-call extract (content + completion-metadata) is explicitly NOT worth porting** — flagged as an anti-pattern by Stagehand's own digest for a single-shot use case (doubles latency/cost for a signal a single schema field could carry). Silver is right to skip this; if Silver ever adds pagination/chunked extraction, a `completed:boolean` field baked into the same schema (one call) is the better port, not Stagehand's two-call split.

## Priority summary
- **HIGH — adopt now:** resolution cache with self-heal (item 1) — biggest relative latency win precisely because Silver is host-brain/no-model, so cache hits skip an entire host round-trip, not just a model call.
- **MEDIUM:** automatic network-quiet gate before snapshot capture (item 2).
- **LOW:** DOM-depth retry ladder (item 3).
- **Do not port:** two-call extract, whitespace-collapsed prompt authoring (Stagehand's own noted code smell), `injectUrls`' silent `?? ""` fallback (Silver already beats this).
