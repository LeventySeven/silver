## 1. What Browserbase does + the mechanism

Browserbase's "session" is a cloud-hosted, four-tier system, NOT a local concept:

1. **`api.browserbase.com` (Fastify)** ‚Äî `POST /v1/sessions` runs an auth path that is deliberately slow: argon2/bcrypt verification of the API key measured empirically at p50 ~830ms, p99 ~941ms (`BROWSERBASE_R2_04...md:310-338`). This 700ms+ auth tax is paid on every session create regardless of allocation speed.
2. **`browser-boss.<region>` (internal, mTLS-only, HTTP 464 from AWS ALB target-group rejection when probed publicly)** ‚Äî the allocator. Maintains a **pre-warmed pool of Firecracker/Kata microVMs, rebuilt every 30 min via cron**. Warm allocation: 300ms‚Äì2s. Cold (new microVM boot + Chrome start): 4‚Äì14s. **Proxies force the cold path** ‚Äî `proxies: true` requires launching Chrome inside the VM with `--proxy-server=...` because pre-warmed pool entries can't have a proxy baked in (`¬ß11`, lines 419-456).
3. **The microVM (ephemeral, one Chrome per VM, destroyed not reused on session end)** ‚Äî fingerprint resolution happens in two layers: (a) *pool-selection* time, OS/locale partition the warm pool (`pool_key = region:os:locale`); (b) *runtime*, viewport/UA/locale are injected post-launch via `Emulation.setDeviceMetricsOverride`, `Emulation.setUserAgentOverride`, `Emulation.setLocaleOverride` ‚Äî no Chrome restart needed (`¬ß12`, lines 460-490). Mobile device emulation is the one exception that forces a cold-Chrome path (needs a launch flag).
4. **`connect.<region>` (Express+helmet on usw2/use1, rewritten to Go on euc1/apse1 for lower per-socket memory ‚Äî Node `ws` ~30KB/socket vs Go goroutine ~4KB)** ‚Äî the WebSocket reverse proxy. Auth is dual-mode and mutually exclusive: `apiKey` (control-plane, SDK convenience) XOR `signingKey` (a **JWE, not JWT** ‚Äî `dir+A256GCM` symmetric encryption, so the client never sees the internal pod IP; payload is `{sessionId, projectId, podHost, podPort, exp}` ‚Äî `¬ß7`, lines 262-280). This is architecturally the crux: connect can route a WS purely from a self-contained encrypted token with no DB lookup.

**Session state machine** (`¬ß14`, lines 518-562): `PENDING ‚Üí RUNNING ‚Üí {COMPLETED | FAILED | TIMED_OUT}`. No `PUT`/`PATCH /v1/sessions/{id}` exists anywhere in the Fastify route table (confirmed via 404-vs-401 route enumeration) ‚Äî **sessions are immutable post-create, append-only**. `TIMED_OUT` fires at >30min idle with no `keepAlive` flag. On any terminal state the microVM is destroyed and a replacement is spawned into the pool ‚Äî reuse across tenants is explicitly avoided as a security property.

**Recording** (R2-07) is a *separate* pipeline from the CDP control plane, and it is undergoing a real architecture pivot the Silver research should note: BB is **deprecating rrweb DOM-event replay** (`GET /v1/sessions/{id}/recording` still returns rrweb JSON inline, `{data,sessionId,timestamp,type}[]`, but is legacy) in favor of **PNG-frame video capture ‚Üí ffmpeg fMP4 ‚Üí HLS**. Mechanism, verbatim from BB's own blog (`¬ß2.3`, lines 195-244): capture is *adaptive-fps* ("if nothing changes for 10 seconds we capture nothing"), frames are PNG not JPEG (lossless + cheap CPU ‚Äî JPEG rejected as "lossy and CPU-intensive", WebRTC rejected because in-browser encoding "steals CPU from the workload"), 10-second segment boundaries, up to 10 tabs captured as separate streams, encoding is deferred/lazy (~8% of sessions ever get replayed, so only the first ~30s is eagerly encoded, rest is just-in-time on first watch), and delivery is HLS with each **tab as a multi-variant playlist entry** (repurposing bitrate-variant semantics for tab-switching) with per-segment CloudFront-signed URLs validated at the CDN edge. Notably ffmpeg's parallel per-segment encoding forces a **manual byte-level patch of the fMP4 `tfdt` box** to fix `baseMediaDecodeTime`, since there's no ffmpeg flag for it (`¬ß2.3` line 224-225, `¬ß7.3` code lines 825-843).

**Live View / debug** (`¬ß4` of R2-07, lines 384-486): `connectUrl` (CDP control, R/W) and `debugUrl`/`debuggerFullscreenUrl` (an HTTPS page wrapping BB's own fork of Chrome DevTools frontend, `browserbase/devtools-frontend`, with a commit literally titled "Remove some screencast UI to enable clean embedding") are **architecturally distinct URLs with distinct auth** (`apiKey` query vs `signingKey` query) even though both ultimately drive the same CDP target ‚Äî a human taking over via Live View and the SDK's own commands literally fight over the same `Input.dispatchMouseEvent` channel. Disconnect is signaled to the parent frame via `postMessage("browserbase-disconnected")`.

## 2. Captcha/stealth ‚Äî LOCAL keyless subset vs cloud/paid (flagged)

**SKIP (cloud/paid, not applicable to Silver):**
- `captcha-boss` ‚Äî a dedicated regional k8s microservice (confirmed via CT-log cert transparency across 4 regions) that routes to a **multi-vendor pool of paid solver APIs**, primary hypothesis CapSolver (medium-high confidence: 5‚Äì30s solve times match CapSolver's 8‚Äì25s tier, founder Paul Klein on-record: "we integrate with a bunch of known vendors... we do some stuff in-house", BB engineer denial of click-farms rules out human-tier 2Captcha/DeathByCaptcha). Token-injection pattern (`g-recaptcha-response` textarea filled directly, page never visually shows solved state) confirms API-vendor integration, not extension-click simulation. **This requires a paid third-party account ‚Äî explicitly out of scope for a keyless CLI.**
- Cloudflare/Stytch "Web Bot Auth" + Fingerprint.com "Authorized Agent Detection" partnerships ‚Äî pre-empt captcha entirely by cryptographically signing BB's bot traffic so partner WAFs whitelist it. This is a **B2B trust relationship**, not a technique Silver can replicate keyless.
- Residential/datacenter proxy assignment (`¬ß11` of R2-04) ‚Äî paid upstream proxy provider, cold-boots Chrome with `--proxy-server`. Skip.
- HLS video recording infra (S3 + CloudFront + ffmpeg worker pool) ‚Äî real infrastructure cost, not relevant to a local single-machine CLI.

**LOCAL KEYLESS WINS for Silver (concrete, adoptable without any vendor key):**

1. **Runtime CDP Emulation overrides without restart, applied AFTER launch** ‚Äî Silver already does `Emulation.setTimezoneOverride`/`Emulation.setLocaleOverride` (`handlers.ts:2202,2217`) but has no `Emulation.setUserAgentOverride` or `Emulation.setDeviceMetricsOverride` command. Browserbase's pattern (`¬ß12`) of separating *pool-selection-time* traits (OS/locale ‚Äî Silver's analog: launch args) from *runtime-injectable* traits (UA, locale, viewport, screen) is directly portable: add a `set useragent <string>` and `set viewport <WxH>` verb using the same `Emulation.*` CDP calls Silver already has working plumbing for. Zero cost, zero keys, immediate fingerprint-diversity win for stealth-conscious tasks.
2. **The immutable append-only session state machine** (`PENDING‚ÜíRUNNING‚Üí{COMPLETED|FAILED|TIMED_OUT}`) is a cheap, load-bearing design pattern Silver's `SessionInfo` doesn't currently express. Silver's `session.ts:37-46` records `idleTimeoutMs` in `OpenOptions` but per the file's own doc comment "unused by open itself" ‚Äî there is no enforcement, no `gc`/reap sweep, and no `TIMED_OUT` concept anywhere in the codebase (`grep` for `TIMED_OUT`/`idleTimeout` enforcement returned nothing beyond the unused field). Browserbase enforces idle-timeout at >30min purely from the session's own heartbeat ‚Äî Silver could adopt an equivalent **local** idle-reaper: check `mtime` on the sidecar or a lightweight heartbeat write on each command, and have `silver session gc` (which the doc comment implies exists conceptually but isn't wired to `idleTimeoutMs`) actually kill+cleanup browsers past their idle budget. This is a correctness gap, not a feature gap ‚Äî the field is dead code today.
3. **Separate control vs. observation URL/channel** ‚Äî Browserbase's `connectUrl` (write) vs `debugUrl` (DevTools UI, human-drivable) distinction is directly reproducible locally and keylessly: Playwright's CDP endpoint already exposes `http://127.0.0.1:<port>/json/version` and the *actual Chrome DevTools frontend* is servable straight from Chromium's own `chrome://inspect` / `devtools://devtools/bundled/inspector.html?ws=<host>:<port>/devtools/page/<targetId>` ‚Äî no fork, no cloud proxy needed. Silver's `session.ts` already resolves `wsEndpoint` and reads `DevToolsActivePort` (`waitForDevToolsPort`, lines 272-286); it could trivially print a ready-to-open local DevTools URL (`silver session inspect <name>`) for human takeover/debugging, giving Silver "Live View" parity with zero new infra ‚Äî literally already 90% present in the CDP handshake code, just not surfaced as a command.
4. **Adaptive-capture idea (screenshot-diffing), not full HLS**: BB's frame-skip-on-no-change heuristic ("if nothing changes for 10s, capture nothing") is a useful *local* recipe for a lightweight `silver session record` that hashes `CaptureScreenshot` output and only writes new PNGs on change, then optionally shells out to local `ffmpeg` (already a common dev-machine binary, no cloud storage needed) to produce an MP4 for debugging/eval-review. This is a scoped, keyless echo of BB's pipeline ‚Äî not the S3/CloudFront/HLS multi-tab-variant stack (which is genuinely cloud-shaped and should be skipped).
5. **JWE-style self-contained session tokens** are irrelevant for a fully local single-user CLI (no remote routing problem to solve) ‚Äî correctly out of scope, noted only to explain why Silver's simpler pid+sidecar model is actually the *right* level of complexity here, not a gap.

**Bottom line:** Browserbase's session/recording/captcha stack is ~90% cloud multi-tenant infrastructure (microVM pools, regional k8s services, paid solver vendors, S3/CloudFront) that is correctly out of scope for a keyless local CLI. The genuinely portable ideas are narrow but real: (1) finish the Emulation-override surface Silver already started (UA + viewport, matching the timezone/locale pattern already in `handlers.ts`), (2) actually enforce the idle-timeout field that already exists but does nothing, and (3) expose the local DevTools live-view URL that Silver's session handshake already resolves internally. All three are small, keyless, and use code paths Silver already has 80%+ built ‚Äî they close real gaps without adding any cloud/vendor dependency.
