# browser-use — Top 5 DX/capability wins vs. every competitor, and the Silver gap

Source read in full: `browser_use/dom/serializer/serializer.py` (1290 lines), `browser_use/tools/registry/service.py` (611 lines), `browser_use/agent/service.py` (4144 lines, `multi_act` region 2719-2838), `browser_use/browser/watchdogs/` (13 files), plus cross-check against `r2-browseruse-controller.md` and `r2-browseruse-dom.md`. Silver comparison read in full: `src/perception/serialize.ts` (292 lines), `src/security/registry.ts` (130 lines), `src/security/redact.ts` (54 lines), `src/actuation/pagechange.ts` (102 lines), `src/actuation/actions.ts` (373 lines), `src/core/session.ts` (494 lines). ## 1. DOM serializer: bounding-box occlusion + synthesized compound components (the single biggest DX-for-the-LLM win in the corpus)

`serializer.py` runs a five-stage pipeline: `_create_simplified_tree` (line 435, recurses shadow DOM/iframes, drops `SVG_ELEMENTS` at line 21-38 as decorative) → `PaintOrderRemover` (occlusion by paint order) → `_optimize_tree` (line 542) → `_apply_bounding_box_filtering` (line 729) → index assignment. The bbox filter drops a child that is ≥99% (`DEFAULT_CONTAINMENT_THRESHOLD`, line 57) contained inside a `PROPAGATING_ELEMENTS` parent (`<a>`, `<button>`, `<div role=button|combobox>`, `<span role=button|combobox>`, `<input role=combobox>`, enumerated line 45-56) — with five carve-out exceptions (form elements, onclick, aria-label, interactive role, nested propagator) so a real nested control inside a link isn't silently eaten. `_add_compound_components` (line 150, called at line 524) synthesizes virtual nodes browser-use's own DOM walk doesn't naturally expose: an `<input type=range>` gets a synthesized `slider` node carrying min/max; `<select>` gets `_extract_select_options` (line 335) — first 4 options plus a `format_hint` (numeric/date/email) sniffed from option value shapes; date/time inputs get an ISO-format hint baked into `_build_attributes_string` (line 1088) specifically so an LLM doesn't guess a locale date format and silently submit a wrong one.

**Why it wins DX over every competitor**: this is the difference between "the model gets raw DOM shape" (AgentQL, most Stagehand modes) and "the model gets pre-digested affordances." Fewer wrong actions per task, fewer retry loops — a direct hit on both cost and reliability, not just token count.

**Silver: confirmed GAP, worth adopting selectively.** `src/perception/serialize.ts` is a flat renderer over `SnapNode[]` from `walk.ts` — no occlusion filtering, no compound-component synthesis, no format-hint injection. A `<select>` or `<input type=date>` in Silver's output is a bare `role name value` line; the host LLM gets no disambiguating hint and must guess formats itself. Silver's per-node format IS leaner (plain text vs. browser-use's attribute-heavy pseudo-HTML), a genuine opposite-axis win worth preserving — but the compound-component + date-hint layer is pure quality-of-output that costs little to port since `walk.ts` already carries the DOM node metadata needed (input type, min/max, option values) to hang synthesis logic off. **Priority: P1** — cheap, isolated addition (new synthesis function called from `serialize.ts`, no architecture change), directly reduces host-LLM wrong-action rate on the two most format-ambiguous control types (select, date/time input).

## 2. Decorator-driven action registry with auto-synthesized Pydantic param models

`Registry.action(...)` (line 291) turns any `async def foo(params: FooParams, browser_session, ...)` into an LLM-callable tool by decoration alone. `_normalize_action_function_signature` (line 75) separates injected "special" params (`browser_session`, `file_system`, `has_sensitive_data`, enumerated in `_get_special_param_types`, line 57) from real action params, and `create_model` (imported line 12, invoked lines 159/162/285/553) auto-generates the Pydantic validation model when the plugin author didn't supply one — so third-party tool/MCP authors get validation, `domains=` scoping, and a `terminates_sequence` flag for free from a plain typed function.

**Why it wins DX**: it's the extensibility mechanism that makes browser-use's own MCP-server mode and community action packs possible without hand-written schemas per action — competitors with hardcoded action sets (AgentQL) can't match this without a rewrite.

**Silver: correctly NOT adopted, by design.** `src/security/registry.ts` is a phase-quarantine registry — a hardcoded `Set<string>` (`READ_ONLY_VERBS` line 24, `ACTOR_VERBS` line 85) gated by `--enable-actions`/`readOnly` (lines 119-122), and `actuation/actions.ts` dispatches a closed `ActVerb` union via switch/case, not dynamic registration. This is intentional: Silver's whole quarantine-as-code security posture depends on the dispatchable verb set being closed and statically auditable — dynamic action registration is the opposite goal and would reopen the exact prompt-injection surface the quarantine model exists to close. **No adoption of the registration mechanism.** The one reusable idea — auto-deriving a param-validation schema from a function/verb signature instead of hand-writing Zod per verb — is a maintainability nicety, not urgent. **Priority: P3 / optional.**

## 3. `multi_act`: two-layer mid-batch page-change abort with partial-results preservation

`agent/service.py:2719-2838`. Before running a queued batch of actions from one LLM turn, it snapshots `pre_action_url`/`pre_action_focus` (lines 2769-2770); after each action, a **static** guard (`registered_action.terminates_sequence`, declared true on `navigate`/`search`/`go_back`, checked line 2806) and a **runtime** guard (URL or focus target actually changed, lines 2812-2818) can independently abort the remaining queue, logging the skip reason. Any exception mid-batch still returns partial `ActionResult`s (lines 2820-2836) instead of losing state; `done` may only be the sole action if not first in a batch (2751-2755).

**Silver: functionally equivalent via a different mechanism, one narrow gap.** `src/actuation/pagechange.ts` computes settle-then-fingerprint (`url + focusedBackendNodeId + domNodeCount`, lines 34, 71-83) and stamps `page_changed`/`stale_refs` as a **flag** returned to the host, per the file's own doc comment: the CLI "must NEVER auto-embed a fresh snapshot... that is the host's decision." Because Silver dispatches one verb per CLI invocation rather than a pre-planned action batch, there's no in-process queue to abort — the equivalent safety property (never act on a stale ref) is enforced per-call by `groundRef`'s generation check in `actions.ts:134-136`, which fails *before* any Playwright call touches a stale/unknown ref. The gap is narrower than the seed's framing suggested: Silver's `batch` verb (cli.ts) chains CLI commands with only `--bail`-on-error, no browser-use-style "stop remaining batched commands once a page-changing one succeeds" logic — it relies on the next command's own ref-check to fail loudly rather than skipping proactively. **Priority: P2** — add a page-changed short-circuit to `batch`, using the same fingerprint pagechange.ts already computes, so a page-changing step in a batch stops the remaining steps instead of letting them fail one-by-one against stale refs.

## 4. Watchdog decomposition + CDP session persistence — the sharpest engine-efficiency signal in this pass

`browser/watchdogs/` (13 files: `dom_watchdog.py`, `crash_watchdog.py`, `downloads_watchdog.py`, `popups_watchdog.py`, `security_watchdog.py`, `permissions_watchdog.py`, `captcha_watchdog.py`, `storage_state_watchdog.py`, `har_recording_watchdog.py`, `recording_watchdog.py`, `screenshot_watchdog.py`, `aboutblank_watchdog.py`, `local_browser_watchdog.py`) subscribe to `bubus` events via `LISTENS_TO`/`EMITS` class attrs. `DOMWatchdog` races DOM-tree build against screenshot capture in parallel (`create_task_with_error_handling`, lines ~356-385), does a bounded 2s pending-network-request probe with timeout fallback (~267-279), and — critically — **caches a `DomService` instance across calls, reusing the CDP client connection** (~546-554; `__del__` deliberately avoids tearing it down). The `BrowserSession` this all sits inside is a long-lived daemon-scoped object across an entire agent run, not per-command.

**Silver: confirmed, empirically, this is the real gap.** `src/core/session.ts`'s own doc comment states it outright (lines 5-7): "Every later command `connect()`s over CDP, does its work, and disconnects — the browser keeps running. Cross-command [state]..." — i.e., Silver's detached-browser architecture means the *browser process* persists but the *CDP session/transport* does not; every single CLI invocation pays a fresh `chromium.connectOverCDP` (line 338) round-trip. This is exactly the "TS + per-command CDP reconnect" gap named in the task brief, and it's here in black and white in Silver's own comments, not inferred. browser-use's watchdog layer isn't the fix by itself — it's architecture-scale over-fit to Silver's single-process-per-invocation CLI model, and the seed's caution against full event-bus adoption stands. But the two extractable, high-leverage pieces are: **(a)** parallel DOM+screenshot capture instead of serial where Silver currently does both per verb, and **(b)** — the actual engine fix — keep a CDP session alive across a `batch` invocation's multiple sub-commands instead of reconnecting per sub-command, since `batch` already runs sub-commands in one process. **Priority: P1 for (b)** — this is the direct, low-risk lever on the latency gap named in the task brief; it doesn't require a persistent daemon (out of scope for a keyless CLI's process model) but removes N-1 reconnects for an N-step batch, which is where most real task latency accumulates.

## 5. `<secret>` placeholder indirection — real value never tokenized into LLM context

`tools/registry/service.py:427-490` (`_replace_sensitive_data`) plus `tools/service.py:781-830`. The LLM only ever sees `<secret>login_pw</secret>` in its own emitted action params; `secret_pattern = re.compile(r'<secret>(.*?)</secret>')` (line 442) resolves it to the real value only at execution time, after Pydantic validation, immediately before the Playwright call. Values are domain-scoped via `match_url_with_domain_pattern(current_url, domain_or_key)` (line 458) against the *current* page URL — a secret registered for `bank.com` cannot resolve on `evil.com` even under prompt injection. `execute_action` (line 365) restricts `sensitive_data` plumbing to actions that declare it, not the full special-context dict.

**Silver: confirmed GAP, highest-priority of the five.** `src/security/redact.ts` (54 lines, read in full) only guards the READ path — `redactValue` (line 44) masks password-shaped/card-shaped values in *outbound* snapshot/get-text output via `PASSWORD_HINT_RE` and `CARD_RE` (lines 29-32). There is no `<secret>`/placeholder/vault mechanism anywhere in `cli.ts` or `actuation/actions.ts` (grep-confirmed) — no WRITE-path equivalent. A `silver fill` targeting a password field takes the literal value as a CLI argument, meaning whatever produced that command line (the host LLM's context, or an env var it echoed) already had the raw credential in view. This is in direct tension with the project's own "keyless, host LLM is the brain" framing: keyless-for-the-model-provider guards a different thing than secret-never-in-model-context, and right now Silver has only the former. **Priority: P0** — add `--secret name=value` / `SILVER_SECRET_<NAME>` env registration, resolve `<secret>name</secret>` inside `fill`/`type` at the actuation choke point (same site `redactValue` already occupies on the read side), and domain-scope resolution against the live page URL exactly as browser-use does. This is the one item where Silver is not just behind a competitor feature-for-feature but is carrying a real security gap relative to its own stated threat model.
