## 1. How Stagehand identifies a node: `encodedId = "{ordinal}-{backendNodeId}"`

CDP `backendNodeId`s are only unique **within one renderer process** (one CDP target/session). A page with a same-origin iframe and a cross-origin (OOPIF) iframe has *two or more* independent backendNodeId spaces that can collide numerically. Stagehand's fix is a compact per-frame **ordinal**, assigned lazily and monotonically on a `Page` instance:

```ts
// page.ts:1087-1098
private ensureOrdinal(frameId: string): number {
  const hit = this.frameOrdinals.get(frameId);
  if (hit !== undefined) return hit;
  const ord = this.nextOrdinal++;
  this.frameOrdinals.set(frameId, ord);
  return ord;
}
public getOrdinal(frameId: string): number { return this.ensureOrdinal(frameId); }
```

Every node's public identity is `${page.getOrdinal(frameId)}-${backendNodeId}` (`capture.ts:384,406`, `a11yTree.ts:96-97`). This single string is what the LLM sees in `[123-4567] button: "Submit"` outline lines (`treeFormatUtils.ts:8-16`) and it doubles as the map key for `xpathMap`, `tagNameMap`, `scrollableMap`, and `urlMap` — one flat namespace across every frame in the page, collision-free by construction. Ordinals persist for the life of the `Page` object and are **preserved across root-frame swaps** (client-side navigation that changes the top frame id keeps the same ordinal — `page.ts:466-469`, `renameNodeId`), so a cached `encodedId` from a prior turn does not silently start pointing at a different frame after a same-document navigation.

## 2. AX node → DOM join: per-session `SessionDomIndex`, not per-frame

Stagehand's core efficiency move is deduplicating `DOM.getDocument` calls at the **CDP session** level, not the frame level, because same-process iframes physically live inside their parent's session/target and calling `DOM.getDocument` once per frame there would be redundant traversal of the same document tree. `buildSessionIndexes` (`capture.ts:314-331`) collapses `framesInScope` down to unique `session.id` values first, then calls `buildSessionDomIndex` (`domTree.ts:257-349`) exactly once per unique session:

- One iterative (non-recursive, stack-based to avoid call-stack blowups on deep DOM) DFS over `DOM.getDocument({depth:-1,pierce:true})`'s result.
- Records, per `backendNodeId`: absolute XPath (`absByBe`), tag name (`tagByBe`, enriched with `input, <type>`), scrollability (`scrollByBe`), which document root it belongs to (`docRootOf` — needed because `pierce:true` flattens *same-process* iframe documents into one tree, so nodes must be partitioned back by owning document), and iframe→contentDocument root mapping (`contentDocRootByIframe`).
- Also records an **Euler-tour DFS enter/exit index** per node (`enterByBe`/`exitByBe`, `domTree.ts:269-270,286-336`) — a monotonically increasing counter stamped on tree-enter and again on tree-exit. This is the mechanism that makes `ignoreSelectors` subtree-exclusion an O(log n) binary-search range check (`capture.ts:668-696`) instead of a tree walk: any node's `enter` index falling inside `[start,end]` of an excluded node is structurally a descendant, no parent-pointer chasing needed.

Per-frame slices (`collectPerFrameMaps`, `capture.ts:341-414`) then just filter this shared index by `docRootOf === docRootBe` — genuinely O(nodes) total across the whole page instead of O(frames × nodes).

**Silver's equivalent (`walk.ts:170-178`)** calls `DOM.getDocument({depth:-1, pierce:true})` exactly **once for the whole page**, which is actually *simpler* than Stagehand's per-session-index scheme for the common same-process case (one call regardless of frame count) — but it has no session partitioning at all, because Silver never opens more than the one default CDP session (`page.context().newCDPSession(page)`, `walk.ts:158`). `pierce:true` descends into same-process iframes' `contentDocument`, which `collectDom` (`walk.ts:398-416`) already walks. So for same-process iframes Silver and Stagehand land in roughly the same place. The divergence is entirely in OOPIF handling (§3).

## 3. Cross-origin iframes: adopted CDP sessions vs "skip silently"

Chrome puts a cross-origin iframe in its own renderer process (OOPIF) once site isolation kicks in — which is the default for basically all real cross-origin embeds (payment widgets, auth iframes, ad frames, YouTube embeds). `pierce:true` on `DOM.getDocument` **does not cross process boundaries** — an OOPIF's contentDocument is simply absent from the pierced tree, and a same-session `Accessibility.getFullAXTree({frameId})` call against it throws (`"Frame with the given id was not found"` class errors).

Stagehand's `Page` explicitly adopts OOPIF child targets via `Target.attachToTarget({flatten:true})` and registers each as its own CDP session (`page.ts:148 "adopted OOPIF sessions"`, `page.ts:519 this.sessions.set(childSession.id, childSession)`); `FrameRegistry.onFrameAttached`/`adoptChildSession` record which `sessionId` owns which `frameId` (`frameRegistry.ts:88-108`, doc-commented at the top as one of the registry's three jobs). `getSessionForFrame(frameId)` (`page.ts:604-607`) then routes every downstream CDP call — `Accessibility.getFullAXTree`, `DOM.getDocument`, `DOM.getFrameOwner` — to the *correct* session automatically. `sessions.ts`'s `ownerSession()`/`parentSession()` wrappers exist specifically so a caller never has to remember "this frame's DOM calls must go to a different session than its parent's."  The result: a cross-origin iframe's accessibility content is captured, encoded (`${ordinal}-${backendNodeId}` using that child session's own backendNodeId space, disambiguated by the ordinal), and spliced into the outline exactly like a same-process iframe (`mergeFramesIntoSnapshot`, `capture.ts:811-870`, using `iframeHostEncByChild` to nest child outlines under the parent iframe's line via `injectSubtrees`).

**Silver's walk.ts is explicit that it does NOT do this.** The doc comment says it outright (`walk.ts:151-152`): *"cross-origin / detached child frames are skipped silently (they simply don't appear)."* Mechanically: `walkFrame()` calls `cdp.send('Accessibility.getFullAXTree', {frameId})` **on the single page-level CDP session** (`walk.ts:158,237-239`); for an OOPIF this throws (wrong-target error) and the `catch` returns `[]` (`walk.ts:241-243`) — the iframe's `Iframe` role node still appears as a leaf in the outline, but everything inside it is invisible to the host model. There is no `Target.attachToTarget`/`flatten` call anywhere in Silver's perception layer, no session registry, no per-frame session routing. This is a **functional gap, not just a token-efficiency one**: any flow that requires clicking/reading inside a cross-origin iframe (Stripe/PayPal checkout widgets, many SSO/auth iframes, embedded video players, some CAPTCHA/consent frames) is currently unreachable through Silver's snapshot+ref grounding, full stop — the host LLM never even sees that the content exists to ask for it.

## 4. Frame-tree topology + XPath composition across frame boundaries

Stagehand additionally builds a persistent `FrameContext` (`capture.ts:141-151`, `buildFrameContext`) capturing `rootId`, a `parentByFrame` index, and the full frame-id list, then walks it parent-first (`computeFramePrefixes`, `capture.ts:742-803`) to compute each child frame's **absolute XPath prefix** — the XPath of the `<iframe>` element that hosts it in its parent's document, resolved via `DOM.getFrameOwner({frameId})` called against the *parent's* session (`parentSession()`, correct even for OOPIFs since `DOM.getFrameOwner` must run on the frame that contains the iframe element). Every relative per-frame XPath is then prefixed (`prefixXPath`, `xpathUtils.ts`) into one global cross-frame XPath space, which is what powers `deepLocator()` (targeting elements by XPath across shadow DOM and iframes, per the project CLAUDE.md) and the scoped-selector fast path (`tryScopedSnapshot`, `capture.ts:162-307`) that walks iframe hops to resolve a selector directly to its owning frame+tail-selector without building the full-page snapshot at all.

Silver has a lighter version of the topology piece only: `resolveChildFrameId()` (`walk.ts:353-366`) resolves one iframe's content frameId via `DOM.describeNode({backendNodeId, depth:1})` on demand, per iframe host node encountered during the DFS, and recurses with `walkFrame(childFrameId, level+1, depth+1)` (`walk.ts:333-339`, capped at `MAX_FRAME_DEPTH=5`). This is fine for producing a correctly-nested outline of same-process iframes but has no absolute-XPath composition step and no `focusSelector`/`ignoreSelectors` cross-frame resolution — Silver's `selectorScope` (`resolveSelectorScope`, `walk.ts:429-455`) is explicitly main-frame-only (`walk.ts:213 "Optional selector scoping (main frame)"`).

## 5. What Stagehand does with the tree that Silver structurally can't (yet)

Two mechanisms tie directly into "backendId maps / iframe encoding" that are worth flagging even though top5 touched pruning generally:
- `resolveFrameDocRootBackendId` (`domTree.ts` via `capture.ts:714-735`) distinguishes "this frame shares its parent's CDP session" (same-process, needs `DOM.getFrameOwner` to find the *content*-document root inside the pierced tree) from "this frame has its own OOPIF session" (its `SessionDomIndex.rootBackend` IS the doc root already) — a branch Silver's single-session model has no need for, but also can't fall back to.
- `makeIsIgnoredBackendNode`/`buildFrameExclusionIntervals` (`capture.ts:549-696`) let `ignoreSelectors` (e.g. "exclude the cookie banner and everything inside any iframe it hosts") recursively exclude nested iframe subtrees by chaining Euler-tour interval lookups across the `childFramesByParent` map (`resolveChildFramesByParent`, `capture.ts:635-666`, itself built via `DOM.getFrameOwner` per child against its parent's session). Silver's `walk.ts` has no `ignoreSelectors`/exclusion concept at all — only the single positive `selectorScope`.

## Gap summary + adoption priority (keyless-compatible; none require a model call)

1. **HIGH — OOPIF/cross-origin iframe capture.** Adopt CDP sessions for cross-origin iframe targets (`Target.attachToTarget({flatten:true})` on the existing Playwright `page.context()`, or via `page.context().newCDPSession()` per discovered OOPIF target) and route `Accessibility.getFullAXTree`/`DOM.getDocument` calls per-frame-session instead of the single page session. This is a correctness gap (content is invisible), not a polish gap — checkout/auth/consent iframes are common in real automation targets. Needs: a small `sessionForFrame(frameId)` registry in `walk.ts`, `CDPSession.session()` sub-session support already exposed by Playwright's `page.context().newCDPSession(target)`.
2. **MEDIUM — Frame-ordinal stable node IDs.** Replace/augment `backendNodeId` alone as Silver's node identity with `${frameOrdinal}-${backendNodeId}`, persisted on a per-Page (per-CLI-process) ordinal map that survives root-frame swaps. Today Silver's `SnapNode.frameId` is a separate field carried alongside `backendNodeId` rather than folded into the ref/id itself — functionally similar but worth confirming refmap.ts's `eN` minting can't collide across frames with identical backendNodeId values from *future* OOPIF support; adopting this now avoids a rework when (1) lands.
3. **MEDIUM — DOM-depth adaptive retry ladder.** Already flagged in top5 item 5; reiterating here because it's the same `domTree.ts` file as the backendId map build (`DOM_DEPTH_ATTEMPTS`/`getDomTreeWithFallback`, `domTree.ts:12-13,138-171`) — a single `depth:-1` `DOM.getDocument` call with no fallback is the same code path that would need touching for (1) and (2), so bundling the three is cheaper than three separate PRs.
4. **LOW — Euler-tour exclusion index for a future `ignoreSelectors`/`-x` flag.** Not urgent absent the feature, but if Silver ever adds selector-based exclusion (banners, ads), the enter/exit DFS-index + binary-search pattern (`domTree.ts:269-336`, `capture.ts:668-696`) is the correct O(log n) primitive to port rather than re-deriving one.
