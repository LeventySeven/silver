## 1. `browse` — a full sibling keyless CLI Silver should treat as the direct competitive baseline (HIGH, informational + several sub-items below)

`packages/cli` is a completely separate product from `packages/core` (Stagehand SDK): an oclif-based CLI (`browse`) that drives local **or** Browserbase-hosted browsers with the exact same shape as Silver — `open`, `snapshot` (returns `@0-5`-style refs), `click @ref`, `fill @ref`, `type`, `press`, `select`, `upload`, `highlight`, `is visible`, mouse/viewport commands, tab management, `network on/off`, `cdp`, `doctor`, `status`, `stop`. Its bundled `SKILL.md` is the literal analog of Silver's own `skill-data/core/SKILL.md`. This means Silver's closest read-for-read competitor on the "keyless browser CLI for agents" thesis already ships, is documented with an agent-facing skill file, and encodes a maturity model Silver should measure itself against line by line. The concrete, non-duplicate deltas below are extracted from reading it, not from the pitch.

**Keyless check:** `browse` itself is not keyless (`--remote` requires `BROWSERBASE_API_KEY`; agent/act/extract calls in `packages/core` require an LLM key) — but `--local` mode and every driver command listed above run with zero API key, and the CLI-level mechanisms extracted in items 2–5 below are 100% keyless.

## 2. Structured `doctor` report: `{checks[], verdict, next, paths}` vs Silver's flat boolean triple (HIGH)

`packages/cli/src/lib/driver/doctor.ts`'s `buildDoctorReport()` returns:
```ts
interface DoctorReport {
  checks: DoctorCheck[];           // each: {name, status: ok|warn|fail|skip, message, fix?, details?}
  next?: string;
  paths: {lock, pid, runtimeDir, socket};
  session: string;
  target?: ConnectionTarget;
  verdict: "ok" | "warn" | "fail";
}
```
Checks run independently (`runtime`, `session`, `daemon`, `browserbase`, `target`, ...), each carrying a **remediation command** in `fix` (e.g. `` `browse stop --session ${session} --force}` ``) and a `details` bag for machine consumption. `browse doctor --json` always exits 0 (agent-safe); the human path exits 1 on `verdict==="fail"`.

Silver's `handleDoctor()` (`silver/src/core/handlers.ts:1490-1518`) returns a flat `{playwright: bool, chromium: bool, uab_writable: bool}` with no per-check message, no fix string, no verdict, no `next` pointer. This is a real regression in diagnosability: a host LLM hitting a failing doctor call today gets three booleans and has to infer what to do; Stagehand's shape hands the fix command directly.

**Concrete Silver change:** restructure `handleDoctor()` in `silver/src/core/handlers.ts` to emit `{checks: DoctorCheck[], verdict, next}` where each existing probe (`playwright` import, `chromium` executable, `~/.silver` writability) becomes a named check with a `fix` string (e.g. chromium-missing → `"npx playwright install chromium"`), and add 1-2 new checks Silver doesn't have yet: session-lock staleness (Silver already has `core/lock.ts` — surface its state here) and CDP-reachability of the currently-running daemon for the target `--session`. **Keyless.** **Priority: HIGH — cheap, pure DX, directly improves host-LLM self-repair loops.**

## 3. Command-not-found: Levenshtein typo suggestion + explicit alias map (MEDIUM)

`packages/cli/src/hooks/command-not-found.ts` + `packages/cli/src/lib/command-suggestions.ts`: a `command_not_found` oclif hook computes a "did you mean" suggestion via (a) an explicit `aliasSuggestions` map for old/common wrong names (`sessions` → `cloud:sessions:list`, `goto`/`navigate` → `open`) checked first, then (b) segment-aligned Levenshtein distance (`fastest-levenshtein`) between the attempted token prefix and every real command id, capped at `maxSuggestionDistance=5` and a per-token threshold of `max(2, floor(len/3))`. Security-relevant detail worth porting verbatim: it works only on a **sanitized token prefix** (`safeTokenPattern = /^[A-Za-z][A-Za-z0-9_-]*$/`), stopping at the first non-command-shaped token — so a URL, selector, or flag value typed after a typo'd verb never leaks into the suggestion message or telemetry.

Silver's `cli.ts` currently has no fuzzy-suggestion path for unknown verbs (only the `META_VERBS` set and dispatch). A host LLM that emits `silver clik @e5` today gets a bare "unknown command" with no correction hint, burning a round-trip.

**Concrete Silver change:** add a small suggestion helper to `silver/src/cli.ts` (or a new `core/suggest.ts`) using the same two-tier approach — explicit alias map for known LLM-guess patterns (`goto`→`open`, `type`→`fill` if that's a common miss) then Levenshtein distance over the real verb table, with the same sanitized-token-prefix guard so argument values never appear in the error string. **Keyless, no new runtime dependency needed** (Levenshtein distance is ~15 lines to hand-roll, avoiding even the `fastest-levenshtein` devDependency). **Priority: MEDIUM — cheap, real host-round-trip savings on typo'd verbs.**

## 4. isTTY-adaptive output format: table for humans, JSON for pipes, by default (MEDIUM)

`packages/cli/src/lib/output.ts`'s `resolveOutputFormat()`: explicit `--json`/`--format` flags win, otherwise `process.stdout.isTTY ? "table" : "json"` — so the exact same command auto-emits human-table output in an interactive terminal and machine-JSON when piped (which is what a host LLM subprocess spawn effectively always looks like). `outputTable`/`formatTable` do column-width-aware, `--wide`-overridable truncation with a `...`-ellipsis convention, useful for any Silver list-shaped output (`session list`, future `cloud`-style listing verbs).

Silver's JSON envelope pattern (`ok()`/`badRequest()` from earlier reads) is already JSON-first for host consumption, so this item is lower-stakes for Silver's *primary* audience — but any Silver commands meant for direct human CLI use (e.g. a hypothetical interactive `session list`/`memory list`) would benefit from the same isTTY branch rather than committing to one format.

**Concrete Silver change:** low priority unless Silver adds more list-shaped human-facing verbs; if/when it does, port `resolveOutputFormat`'s `isTTY` branch and the truncate-with-ellipsis table formatter as a shared `core/output.ts` helper. **Keyless. Priority: LOW-MEDIUM.**

## 5. Browse.sh — a site-specific "skills" catalog callable from the CLI itself (NOVEL, HIGH long-term)

The bundled `SKILL.md` documents `browse skills list|find|add` against **Browse.sh**, an external catalog of *site-specific* automation recipes keyed `<domain>/<task>` (e.g. `yelp.com/extract-reviews-2ikb22`). Each entry encodes a proven strategy for one task on one website — "API endpoints, selectors, anti-bot handling" — tagged with a `method` (`api`/`fetch`/`browser`/`hybrid`) and install-count as a trust signal. The SKILL.md explicitly instructs the host agent to search the catalog *before* automating a named site by hand, and to broaden the query (domain → site name → task keyword → synonym) rather than give up on an empty result.

This is the most structurally novel idea in this pass and does not map onto anything either prior Silver deepdive digest flagged: it's a **shared, versioned recipe registry** that turns "figure out this site's DOM/selectors/anti-bot quirks from scratch every session" into "look up a known-working recipe first." For Silver, whose entire selling point is host-LLM-driven exploration with no model calls of its own, an equivalent local/pluggable recipe store is a categorical capability, not a DX polish item: it converts repeat-site automation from O(exploration) to O(lookup), and — because Silver already has `silver/src/memory/store.ts` (grep-first episodic markdown) and `silver/src/task/store.ts` (plan/log persistence) — Silver already owns the *storage* primitives, just not a **cross-run, cross-task, keyed-by-site** index over them.

**Concrete Silver change:** add a `silver recipe` (or extend `memory`) subcommand namespace that indexes past successful task runs by `hostname` (from `task/store.ts` logs) and exposes `silver recipe find <domain>` returning the last N successful ref/selector sequences for that domain, purely as a local on-disk cache (no network catalog needed for v1 — that's Browserbase's hosted differentiator, not something Silver should reimplement as a network service). This reuses `task/store.ts`'s existing plan/log format; the new piece is just a `domain → [taskIds]` index file. **Keyless.** **Priority: HIGH as a differentiation lever, MEDIUM as near-term engineering effort** (real design work: recipe staleness/self-heal, not just indexing).

## 6. Per-request network capture dumped to disk, keyed by session (HIGH, keyless, concrete)

`packages/cli/src/lib/driver/network-capture.ts`'s `NetworkCapture` class: `Network.enable` on the page's main-frame CDP session with explicit buffer caps (`maxResourceBufferSize: 5_000_000, maxTotalBufferSize: 10_000_000`), then per-request-id bookkeeping across `requestWillBeSent`/`responseReceived`/`loadingFinished`/`loadingFailed`, writing **one directory per request** (`{counter}-{METHOD}-{domain}-{firstPathSegment}/{request.json, response.json}`) under a session-scoped runtime dir, sanitizing hostnames/paths for filesystem safety (`sanitizeForFilename`). Base64 bodies are truncated to a 100-char preview rather than dumped whole. `browse network on|off|path|clear` exposes this as four commands. Zero model involvement — purely CDP + filesystem.

Silver has no equivalent: no `network` verb, no per-request capture-to-disk facility. This is a genuinely useful capability for a host LLM debugging "why did this form submit fail" or "what API did this SPA call" — currently the host has to either read the page's rendered DOM (useless for XHR-driven state) or Silver would need a bespoke `eval`-based fetch hook.

**Concrete Silver change:** add `silver network on|off|path|clear [--session]` backed by a small `NetworkCapture`-equivalent class in a new `silver/src/actuation/network.ts`, writing under the existing per-session runtime dir (`core/nsdirs.ts` already manages session-scoped paths). Cap buffer sizes identically. This directly composes with Silver's existing "host reads files, not a live UI" model — `network path` returns the directory, host `Read`s the JSON files it cares about. **Keyless. Priority: HIGH — closes a real observability gap with a small, self-contained module.**

## 7. Domain policy enforced at the CDP `Fetch` interception layer, not just at the navigation call site (HIGH — security-relevant gap)

`packages/core/lib/v3/understudy/domainPolicy.ts` builds `Protocol.Fetch.RequestPattern[]` (allowlist → `http(s)://*/*` catch-all + per-request evaluation; blocklist → explicit per-host/per-scheme/per-port patterns) and validates hostnames strictly (label-regex, wildcard only as leading `*.`, rejects any URL-shaped value with `://`, `/`, `:`, `?`, `#` embedded — i.e. it forces domain-only patterns, closing the "sneak a path/query into the allowlist pattern" footgun). Crucially, this policy is designed to be wired into **`Fetch.enable`**, which intercepts *every* subresource request the page makes (scripts, XHR/fetch, images, iframes, websockets-upgrade-adjacent) — not just top-level navigations.

Silver's `security/egress.ts` (read in full) is a well-designed denylist-by-default guard with SSRF/DNS-rebinding protection and suffix-match allowlist hardening — but its own doc comment says it is invoked "on the lowest navigation primitive it controls" and a `grep` across `core/handlers.ts`/`core/session.ts` confirms it is only called from navigation call sites (`page.goto`, redirect-hop re-checks), never wired to a CDP `Fetch.enable`/`Network.setBlockedURLs`-style subresource interceptor. This means Silver's `--allowed-domains` hardening currently governs **where the page can navigate**, but not **what the page can fetch once loaded** — a page on an allowed domain can still exfiltrate data via an `<img src>`/`fetch()`/`beacon` to a disallowed host, and Silver's own egress guard does not see it.

**Concrete Silver change:** extend `security/egress.ts`'s existing `EgressOptions`/`allowedDomains` model with a CDP-level enforcement path: when `--allowed-domains` is set, call `Fetch.enable` (or the cheaper `Network.setBlockedURLs` for a pure blocklist) on session start and continue/fail requests per the same suffix-match logic already implemented, reusing `domainPolicy.ts`'s pattern-generation approach (`toAllowlistFetchPatterns`/`toBlocklistFetchPatterns`) almost verbatim — it's a pure function, trivially portable without pulling in any Stagehand runtime code. **Keyless.** **Priority: HIGH — this is a genuine security hardening gap, not a nice-to-have, given Silver's stated threat model (red-team S5/S6 already named in `egress.ts`'s own doc comment) explicitly includes exfiltration.**

## 8. DeepLocator `>>` hop notation — a compact selector DSL for shadow-DOM/iframe chains (MEDIUM)

`packages/core/lib/v3/understudy/deepLocator.ts`: `resolveLocatorTarget()` splits a selector string on `>>` into hop segments, resolving each through a `FrameLocator` chain (`page.deepLocator('iframe#ifrA >> #btn')`), and separately `resolveDeepXPathTarget()` walks a raw absolute XPath token-by-token, detecting `<iframe>` steps via `IFRAME_STEP_RE` and automatically splitting the XPath into per-frame segments composed through nested `FrameLocator`s — so a single XPath string that crosses N iframe boundaries resolves without the caller needing to manually hop frame-by-frame. `DeepLocatorDelegate` wraps this as an awaitable-free chainable object mirroring the full `Locator` API (`click`, `fill`, `highlight`, `setInputFiles`, `nth`, ...).

This is a genuinely different targeting mechanism from what perception-digest's XPath-prefix-composition item covered (that was about the accessibility-tree *outline* representation; this is about a **caller-facing selector string format** for direct targeting without going through a snapshot at all — useful when a host already knows the exact XPath, e.g. from a cached recipe per item 5, and wants to act without paying for a fresh full-page snapshot).

**Concrete Silver change:** Silver's actuation verbs currently accept `@eN` refs (post-snapshot) or presumably raw CSS selectors for some verbs — worth confirming Silver has no equivalent for "XPath that crosses an iframe boundary" as a first-class selector format. If not, add `>>`-hop and iframe-aware-XPath parsing as an alternate selector syntax alongside `@eN` refs in `actuation/resolve.ts`, reusing the existing frame-topology walk already built for perception (per `walk.ts`'s `resolveChildFrameId`). **Keyless.** **Priority: MEDIUM — valuable specifically as the resolution-side complement to a future recipe/cache feature (item 5), lower value standalone since Silver's snapshot+ref flow already covers the common case.**

## 9. `AgentCache` — whole-trajectory replay cache, distinct from per-action `ActCache` (MEDIUM, extends prior digest's cache item)

`packages/core/lib/v3/cache/AgentCache.ts` is a **separate** cache subsystem from the `ActCache` the extract-digest already covered — this one caches an entire recorded `agent.execute()` run as a typed step sequence (`AgentReplayActStep | AgentReplayFillFormStep | AgentReplayGotoStep | AgentReplayKeysStep | AgentReplayNavBackStep | AgentReplayScrollStep | AgentReplayWaitStep`), not a single action. It records live during execution (`recording: AgentReplayStep[] | null`) and persists via the same `CacheStorage` abstraction. Two details worth porting independent of the replay mechanism itself:
- **`SENSITIVE_CONFIG_KEYS = new Set(["apikey","api_key","api-key"])`** — an explicit redaction set scrubbed before any config object is written to a cache file, so a cached trajectory can never leak a credential that happened to be in the agent's runtime config.
- **`cloneForCache()`/`safeGetPageUrl()`** (`cache/utils.ts`, referenced) — defensive cloning before persistence (same aliasing-bug defense pattern as `CacheStorage`'s `jsonClone()` already flagged in the extract digest) and a URL-getter that can't throw on a torn-down page.

**Concrete Silver change:** if/when Silver's `task/store.ts` gains any form of full-run replay-from-cache (as opposed to its current crash-recovery-log role), the sensitive-key-redaction set and defensive-clone-before-write pattern should be lifted directly — Silver already redacts passwords/cards at the *serializer* (per its own skill blurb), but a future trajectory cache would need the same discipline applied to whatever config/variable bag accompanies a cached run. **Keyless.** **Priority: MEDIUM — not urgent absent a trajectory-cache feature, but cheap insurance to bake in from day one if that feature is built (don't let credentials leak into a cache file that item 5's recipe store would also read).**

## 10. `LifecycleWatcher` — a dedicated navigation state machine, not a bare `waitForX` promise (MEDIUM)

`packages/core/lib/v3/understudy/lifecycleWatcher.ts`: one watcher instance per navigation, explicitly tracking `currentLoaderId`/`expectedLoaderId`/`initialLoaderId` to distinguish "the page I navigated to" from "a different navigation that started before mine finished" (`pendingFollowupNavigation`), with a `waitForNetworkIdle` loop that **restarts** if a follow-up navigation is detected mid-wait (`shouldRestartAfterFollowup`) rather than resolving against a stale loader. It also races an explicit `abortPromise` so a torn-down session cancels the wait cleanly instead of hanging until timeout. This is materially more robust than "wait for network idle once" — it specifically handles the common real-world case of a page that redirects itself (meta-refresh, JS `location.replace`, an SSO bounce) mid-load, where a naive single-shot wait would either resolve too early (against the pre-redirect page) or hang past the real settle point.

Silver's `actuation/wait.ts` (referenced from prior digest) exposes `networkidle` as a caller-opt-in spec but — per the extract digest's finding #2 — has no automatic pre-snapshot gate and, per this pass, no loader-id/follow-up-navigation tracking at all: a click that triggers a client-side redirect could cause Silver's wait to settle against the wrong intermediate page state.

**Concrete Silver change:** when implementing the network-quiet auto-gate already flagged HIGH-priority in the extract digest, additionally track the CDP `Page.frameNavigated`/`Page.navigatedWithinDocument` loader-id sequence for the main frame and restart the idle-wait if a new navigation supersedes the one being waited on — folding this into the same PR rather than shipping a naive single-shot wait that would need a rework later. **Keyless.** **Priority: MEDIUM — bundle with the extract digest's HIGH-priority network-quiet-gate item rather than a standalone task.**

## 11. File-upload input normalization: path-or-buffer, validated, both close to identical (LOW, confirm-only)

`packages/core/lib/v3/understudy/fileUploadUtils.ts`'s `normalizeInputFiles()` accepts a string path (resolved against a `baseDir`, `fs.stat`-validated as a real regular file, not a directory or special entry, then read into a `Buffer`) **or** an in-memory `{name, mimeType, buffer, lastModified}` payload, uniformly producing `NormalizedFilePayload[]`. The validation (reject non-file entries with a named error rather than a generic ENOENT) is the only piece worth flagging as a possible gap-check.

**Concrete Silver change:** confirm `silver`'s `upload` command (`silver/src/actuation/*`) validates the source path is a regular file before attempting the CDP `DOM.setFileInputFiles` call, and surfaces a named, actionable error (not a raw ENOENT) if not — this is a 10-minute defensive check, not a new feature. **Keyless.** **Priority: LOW — verify-and-patch-if-missing, not a build item.**

## 12. `--auto-connect` vs `--local` vs `--remote` vs `--cdp <port|ws-url>` — an explicit target-selection flag matrix with fallback guidance baked into the skill file (MEDIUM, DX pattern)

The `browse` SKILL.md is explicit and prescriptive about *when* to use each browser-target mode, including a documented failure→fallback path: "If `--local` fails with 'No Chrome or Chromium found' and `BROWSERBASE_API_KEY` is set, switch to `--remote` — do not retry `--local`," and "Never retry a failing command unchanged... Init failures are cached for several seconds, so instant retries return identical errors." This is a deliberate anti-loop instruction baked directly into agent-facing docs, addressing a known LLM failure mode (blind retry loops) at the documentation layer rather than only in code.

Silver's own `SKILL.md`/CLI docs should be audited for the same explicit "don't retry X, do Y instead" guidance wherever a Silver command has a known transient-vs-persistent failure distinction (e.g. `ref_stale` — already correctly says "re-snapshot, never guess" per `core/errors.ts`'s doc comment — but check whether `navigation_blocked`, `element_obscured`, and lock-contention errors carry equally explicit non-retry guidance in the skill text, not just the error taxonomy).

**Concrete Silver change:** audit `skill-data/core/SKILL.md` against this pattern; add one-line "don't retry unchanged, do X" guidance for every error class where blind retry is a known failure mode, mirroring the specificity of `browse`'s troubleshooting section. **Keyless.** **Priority: MEDIUM — cheap doc-only change with real reliability payoff for autonomous host-LLM loops.**

## 13. Clipboard already ported, but missing the `Browser.grantPermissions` scoping-to-origin nuance (LOW, confirm-only)

`packages/core/lib/v3/understudy/clipboard.ts`'s `grantClipboardPermissions()` scopes the CDP `Browser.grantPermissions` call to the **current page's precise origin** (`this.originForPage(page)`, `http(s)` only, `undefined` for anything else — permission grant silently skipped rather than attempted against a non-http origin) before every clipboard op, and separately calls `Page.bringToFront` + a `window.focus()` `Runtime.evaluate` to work around clipboard APIs requiring page focus/user-gesture context. Silver's `handlers.ts:1899-1927` clipboard implementation (confirmed already present, not a gap) calls `grantPermissions` with `safeOrigin(page.url())` — structurally the same idea. Worth a fast diff-check only: does Silver's `grantPermissions` call happen on **every** clipboard op (guarding against navigating to a new origin between grants) or once at session start? Stagehand's per-call re-grant is the safer pattern if the current page's origin can change between clipboard calls.

**Concrete Silver change:** confirm `silver/src/core/handlers.ts`'s clipboard handler re-grants per-call (not cached from an earlier origin); if it caches, fix to match Stagehand's per-call re-scope. **Keyless.** **Priority: LOW — likely already correct, quick verify.**
