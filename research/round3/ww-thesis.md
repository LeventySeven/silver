# webwright thesis vs moxxie — "a terminal is all you need" / code-as-action / run-mode

Source: microsoft/webwright (`/Users/seventyleven/Desktop/moxxie/reference/webwright/{README.md,src}`)
Moxxie anchors: `skill/agent-browser/src/cli.ts`, `skill/agent-browser/src/core/handlers.ts`, `skill/agent-browser/src/core/session.ts`

## Thesis recap

Webwright's core bet, stated explicitly in its own README comparison table (`README.md:71-76`):
"agent-browser (Vercel)" — "CLI tool that *another* agent calls," "Discrete subcommands (`open`, `click @e2`, `snapshot`, `eval`)," "One CLI invocation per micro-step" — is a **named row in their own table, distinct from Webwright**. Moxxie is architecturally that "agent-browser" row: `cli.ts` dispatches one verb per process invocation (`run()` at `cli.ts:48`), and `handlers.ts` documents the model explicitly: "The browser-as-daemon model (session.ts): each handler `connect()`s over CDP... Cross-command state... lives in a per-session `moxxie-state.json` sidecar" (`handlers.ts:9-14`).

Webwright instead: (1) separates the agent from the browser — the browser is disposable, the **workspace** (code + logs + screenshots on disk) is the persistent state (`README.md:38,75`); (2) forces each task into **one re-runnable Python script**, `final_script.py`, optionally "crafted" into a parameterized reusable CLI tool via `/webwright:craft` (`README.md:255-256`); (3) writes a full on-disk trajectory per step — `step_NNNN.json`, `steps.md`, `command_history.sh` — so the run is inspectable and literally re-executable after the fact (`agents/default.py:98-176`, `environments/local_workspace.py:133-151`).

These are two different bets for two different actors. Webwright's actor is a **coding model** that writes and debugs Python; moxxie's actor is a **host LLM that emits discrete tool calls** (Claude Code, etc.) with moxxie doing grounding/security, not code generation. The lens question is not "should moxxie become Webwright" (it structurally can't stay keyless if it does — see skip below) but "what re-runnability/observability wins does Webwright get from disk-first design that moxxine can absorb without adding a code-exec surface."

## Gap-alignment findings

1. **No re-runnable command history exists in moxxie at all** (P0, adopt, keyless).
   Webwright's `environments/local_workspace.py:133-143` (`_persist_step_command`) appends every executed command, verbatim, to `command_history.sh` — a literal re-runnable script of the whole session. Moxxie's session state (`handlers.ts:83-110`, `loadState`/`saveState`/`patchState`) tracks only `generation`/`prevTree`/`fingerprint`/`extract` — it never records the verb+args that produced that state. There is no way today to reconstruct "what sequence of moxxie invocations got this session into its current state," so a host agent cannot replay, audit, or hand off a session. Change: in `handle()` (`handlers.ts:157`), after a successful mutating verb (open/goto/act verbs/wait/extract), append a line (`moxxie <verb> <args> [--flags]` reconstructed from `flags`) to `~/.moxxie/sessions/<name>/history.sh`, mirroring `_persist_step_command`. Purely a file append — no model call, no code execution by moxxie itself.

2. **No exportable session transcript / replay artifact** (P0, adopt, keyless).
   Webwright's `final_script.py` + `/webwright:craft` (README.md:255-256) turn a completed run into a durable, reusable artifact the host (or a different agent) can rerun with different parameters. Moxxie has no equivalent — once a session closes, the sequence of actions is gone (only the final `moxxie-state.json`/`refmap.json` survive, and those are grounding caches, not history). Change: add a new meta verb `moxxie session export [--format sh|jsonl]` in `handleSession()` (`handlers.ts:779-802`) that reads the history log from finding 1 and prints it as a shell script (`moxxie open <url>` ... `moxxie click @e3` ...) or JSONL trajectory. This directly ports Webwright's "browsing history is a single re-runnable file" thesis into moxxie's own dispatch shape — no free-form code, no code-exec, just replaying moxxie's own verbs.

3. **No structured per-step debug/trajectory log for post-hoc inspection** (P1, adopt, keyless).
   `agents/default.py:98-176` (`_write_debug_step_artifact`) writes one JSON per step (thought, action, raw output, observation) plus a human-readable `steps.md`. Moxxie's envelope is printed to stdout and then thrown away — the CLI is stateless per-invocation with zero optional logging (`cli.ts` `print(env, json)` at line 154, no log sink). For long agent-driven sessions this makes debugging a stuck/looping host agent hard from moxxie's side alone. Change: add an opt-in `--log-dir <path>` flag (parsed in `flags.ts`, consumed in `withConnection`/`handle` in `handlers.ts:157`) that appends `{verb, args, envelope, ts}` as a JSONL line per invocation to `<log-dir>/moxxie-trajectory.jsonl`. Off by default so it doesn't touch the no-leak/no-side-effect default posture.

4. **Workspace-as-state vs browser-as-daemon: no change needed — moxxie's design is the correct one for its actor, keep it** (P2, skip-cargo-cult).
   Webwright's "browser is disposable, workspace is state" (`README.md:38-42,75`) is optimized for a coding-model actor that can lose/relaunch a browser cheaply because the *code* is durable. Moxxie's actor is a host LLM issuing discrete verbs with no code to fall back on — for that actor the persistent CDP-connected daemon (`session.ts:1-12`, `handlers.ts:120-141` `ensureConnected`/`withConnection`) is load-bearing: it's what lets `snapshot` → `click @eN` → `snapshot` work as cheap round trips without re-navigating. Do not adopt Webwright's disposable-browser model; it would break moxxie's ref-grounding contract (refs are tied to a live page + generation, not to a script).

5. **Full free-form code-as-action (bash/python execution mode) — explicitly do NOT adopt** (skip-cargo-cult, keyless-incompatible as-is).
   Webwright's entire action space is "Free-form Python (writes Playwright scripts itself)" executed via `subprocess.run(command, shell=True, ...)` (`environments/local_workspace.py:176-204`, `execute()`). This is the mechanism that makes their paradigm work, but it is fundamentally a **sandboxed arbitrary-code-execution surface** — the opposite of moxxie's design center: `cli.ts` documents the phase-quarantine registry gate as "the choke point" (`cli.ts:9-14`) and `handlers.ts` documents ref-grounding + egress + neutralization as the security moat (`handlers.ts` `assertNavigable`, `neutralize`/`capOutput`, `requiresConfirm`). A `moxxie run <script>` verb that shells out to arbitrary host-supplied code would bypass every one of those gates (registry, confirm-gate, egress allowlist, refmap grounding) in one step. If ever considered, it would need to be re-expressed as: the host runs its own code with its own sandbox, and only calls moxxie's discrete verbs from inside that script — which is just... the status quo. Flag explicitly: no code-run mode, ever, without re-deriving the entire security model.

6. **`self_reflection` / judge-gated `done` and `image_qa` tool — model calls, invalid as moxxie features** (skip, not keyless).
   `agents/default.py:202-268` (`_self_reflection_gate_error`/`_tool_gate_error`) blocks task completion until a **separate LLM judge call** scores `predicted_label == 1`, and `tools/image_qa.py` (referenced in README `tools/` map) is a vision-model call over screenshots. Both require moxxie to call a model — hard-disallowed. The keyless analogue moxxie already has is the `page_changed`/`stale_refs` fingerprint contract stamped on every action response (`handlers.ts:424-435`) — that is moxxie's correct, keyless version of "did this actually work," and is sufficient; no judge-gate should be added.

7. **History-compaction via LLM summarization call — not keyless, but the *problem* it solves (context growth from repeated large snapshots) is real and has a keyless answer moxxie already half-has** (P1, align).
   `agents/default.py:303-339` (`_compact_history`) and `keep_last_n_observations`/`_prune_old_observation_aria_snapshots` (`agents/default.py:275-301`) exist because Webwright's transcript keeps every full ARIA snapshot inline and it must periodically summarize via a model call to stay in budget. Moxxie avoids the root cause differently and better: `handleSnapshot` (`handlers.ts:308-345`) already does diff-when-shorter observation via `observe(prev?.prevTree, text)` (`perception/diff.js`) so repeat snapshots are not resent in full. Recommendation: keep this design (it already beats Webwright's approach because it needs no model call), but make sure the diffing behavior is documented in `handleSkill()`'s short/full text (`handlers.ts:858-879`) as an explicit selling point ("moxxie's snapshot output is diffed against the prior snapshot, so re-observing a page costs less context than a fresh snapshot" ) since it's currently only implicit in code comments, not surfaced to the host agent reading `moxxie skill`.

8. **Task Showcase / trajectory-viewer dashboards — out of scope for moxxie itself** (P2, skip-cargo-cult).
   `assets/task_showcase/` (Flask dashboard rendering `task.json`+`report.json`) and `assets/compare_trajectory/` (README.md:349-362) are presentation layers over Webwright's own run artifacts. Moxxie is a CLI library, not a run-orchestration product; if the host wants a dashboard over moxxie's `session export`/`--log-dir` output (findings 1-3), that's the host's job, not moxxie's. Do not build a bundled viewer.

9. **Per-step auto-screenshot capture into a session directory** (P2, adopt selectively, keyless).
   `local_workspace.py:106-131,241-273` auto-tracks `recent_screenshots`/`latest_screenshot` per step so debugging a failed run doesn't require the model to remember to call a screenshot tool. Moxxie's `handleScreenshot` (`handlers.ts:372-381`) only captures on explicit host request. Low-cost, low-priority addition: when `--log-dir` (finding 3) is set AND an action's `page_changed` is true, optionally auto-save a screenshot into `<log-dir>/screenshots/`. Purely file I/O, opt-in, no behavior change to the default path.

## Top recommendation

Ship a keyless "replayable session" pair: (a) an append-only `history.sh`-style command log written on every mutating verb (finding 1), and (b) a `moxxie session export` verb that serializes it into a re-runnable script/JSONL (finding 2). This is the direct, keyless-safe translation of Webwright's "the browsing history is a single re-runnable file" thesis onto moxxine's discrete-verb architecture — it captures the re-runnability value without adopting Webwright's arbitrary-code-execution surface (finding 5, hard skip) or its model-gated completion checks (finding 6, hard skip).
