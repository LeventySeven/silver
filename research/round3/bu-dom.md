# browser-use DOM serializer vs moxxie walk.ts ŌĆö gap alignment (round 3, lens: DOM serializer + selector-map + viewport/visibility)

Source read: `reference/browser-use/browser_use/dom/serializer/{serializer.py,clickable_elements.py,paint_order.py}`, `dom/views.py` (EnhancedDOMTreeNode.is_actually_scrollable, is_visible).
Moxxie read: `skill/agent-browser/src/perception/walk.ts` (snapshotNodes, SCAN_JS, SnapNode), `perception/serialize.ts` (render, new-node marking), `perception/refmap.ts`.

moxxie's walk.ts is a single CDP session doing: in-page cursor/hidden scan (SCAN_JS) ŌåÆ `DOM.getDocument` ŌåÆ `Accessibility.getFullAXTree` ŌåÆ DFS join, computing `refEligible` from AX role + a cursor-interactive cascade. browser-use's `DOMTreeSerializer` operates on an already-built `EnhancedDOMTreeNode` tree (DOM+AX+Snapshot merged upstream by `DomService`/`dom_watchdog.py`) and does simplify ŌåÆ paint-order removal ŌåÆ bbox-containment filtering ŌåÆ interactive-index assignment ŌåÆ string serialize, with a much heavier feature set aimed at multi-provider LLM consumption.

## Findings

### 1. [P0, adopt] No cross-frame / iframe AX walk ŌĆö `frameId` is hardcoded, never populated
- **source_does**: `serializer.py::_create_simplified_tree` explicitly special-cases `IFRAME`/`FRAME` nodes and recurses into `node.content_document.children_nodes` (line ~483-490), so form fields inside same-origin iframes (payment widgets, embedded signup forms, cookie-consent iframes) show up in the selector map with their own backend_node_id. `clickable_elements.py` also gates interactivity for iframes on size (>100x100) so tiny iframes aren't wastefully indexed.
- **moxxie_current**: `walk.ts` `SnapNode.frameId` is typed as `string` but every node is stamped `frameId: 'main'` unconditionally (walk.ts:245). `Accessibility.getFullAXTree` and `DOM.getDocument` are called once on the top-level `CDPSession` obtained via `page.context().newCDPSession(page)`; there is no per-frame session, no `Target.setAutoAttach`, no recursion into `contentDocument` for the AX pass (DOM pass does recurse into `node.contentDocument` in `collectDom`, but that only builds the attribute map ŌĆö AX nodes for iframe content are never fetched, so those DOM nodes never get a corresponding AX node and are pruned/invisible). `grep` confirms no other file in `src/` mentions `iframe`/`frameId`/`setAutoAttach` except walk.ts and refmap.ts (which just stores the string).
- **recommendation**: adopt
- **change**: In `snapshotNodes` (walk.ts), after the top-level AX pass, enumerate iframe elements (`nodeName === 'IFRAME'` in `domByBackend`), for each same-origin one open a CDP session on its frame (`page.frames()` lookup by matching `frameId`/URL, or `cdp.send('Page.getFrameTree')` to get frame IDs, then `Accessibility.getFullAXTree({frameId})` ŌĆö CDP supports scoping AX tree fetch to a frame via the `frameId` param in modern Chrome), run the same DFS over that subtree, and splice the resulting SnapNodes in as children of the iframe's SnapNode with the real `frameId` set (not `'main'`). Skip cross-origin (OOPIF) iframes for now ŌĆö those need `Target.setAutoAttach` + a session per target, real added complexity; gate that behind a size heuristic like browser-use's >100x100 check so tiny tracking/ad iframes aren't walked.
- **keyless_ok**: true
- **priority**: P0
- **evidence**: browser-use `dom/serializer/serializer.py:483-490` (IFRAME recursion), `dom/serializer/clickable_elements.py` iframe size gate; moxxie `perception/walk.ts:245` (`frameId: 'main'` hardcoded), `perception/walk.ts:122` (single `newCDPSession`), `perception/walk.ts:285-303` (`collectDom` recurses `contentDocument` for DOM attrs only, not paired with an AX fetch).

### 2. [P0, adopt] No occlusion / paint-order check ŌĆö modal-covered elements still surface as clickable
- **source_does**: `paint_order.py::PaintOrderRemover` maintains a rectangle union per z-order layer and marks nodes `ignored_by_paint_order = True` when a later-painted (higher stacking) element's box fully covers them, so an element sitting under a modal backdrop is excluded from the selector map even though its CSS visibility is "visible".
- **moxxie_current**: `SCAN_JS` (walk.ts:381-438) only checks `style.visibility === 'hidden' || style.opacity === '0'` and a parent-cursor heuristic (walk.ts:409-412: if a wrapper has `cursor:pointer` but the element itself has no direct interaction signal, skip if the *immediate* parent also has `cursor:pointer`). There is no hit-testing against what's actually painted on top ŌĆö a page with an open modal (`position:fixed` overlay + backdrop) leaves the underlying page's buttons `visibility:visible`, non-zero-size, and AX-included, so moxxie will happily mint refs for elements the agent literally cannot click (Playwright's actual click would either hit the backdrop or throw "element is not visible/intercepted").
- **recommendation**: adopt (heuristic version, not full rect-union)
- **change**: In `SCAN_JS` (walk.ts), for each candidate element with a non-empty `rect`, add a cheap top-of-stack check: `var cx = rect.left + rect.width/2, cy = rect.top + rect.height/2; var topEl = document.elementFromPoint(cx, cy); var occluded = topEl !== el && !el.contains(topEl) && !topEl.contains(el);` and thread `occluded` through the record so `visit()` in walk.ts can either drop the node from `refEligible` or (safer) keep it in the tree but flag `flags.occluded = true` and exclude it from `refEligible` so the agent can't blind-click it, while a note is still visible for debugging. This reuses the rect already computed in SCAN_JS ŌĆö no full rect-union bookkeeping needed since only viewport-visible clickable candidates need it (elements scrolled off-screen already have `elementFromPoint` return something else, which is fine ŌĆö moxxie doesn't restrict itself to viewport, so gate this check to only run when the element's rect intersects the current viewport bounds, otherwise skip the check and trust AX/visibility as today).
- **keyless_ok**: true
- **priority**: P0
- **evidence**: browser-use `dom/serializer/paint_order.py` (RectUnionPure, `_MAX_RECTS` cap, `contains`); moxxie `perception/walk.ts:393` (`isHidden` only checks `visibility`/`opacity`), `walk.ts:409-412` (parent-cursor heuristic is the only occlusion-adjacent logic present).

### 3. [P1, adopt] No compound-component synthesis for `<select>` / date-time / range / file / number inputs
- **source_does**: `serializer.py::_add_compound_components` + `_extract_select_options` walk a `<select>`'s DOM children directly and inject a synthetic `options=...` attribute with the first 4 option labels + a `count=N` + a `format_hint` (numeric / country-code / date / email) inferred from the option values (lines 271-298, 335-415). For `<input type=date|time|...>` it injects an explicit `format=YYYY-MM-DD` attribute (lines 1102-1177) because AX/placeholder frequently omit it. Range inputs get synthetic min/max, file inputs get a synthetic "Browse Files"/"File Selected" pair with current filename.
- **moxxie_current**: `walk.ts` exposes `role` (likely `combobox`/`listbox` for select, from `INTERACTIVE_ROLES`/`CONTENT_ROLES` in roles.ts) and `flags.placeholder` from the `placeholder` attribute only (walk.ts:229). There is no reading of a select's `<option>` children, no date/time format hint, no range min/max surfacing beyond whatever the AX tree natively exposes (which for a plain unstyled `<select>` is often just the currently selected option's accessible name ŌĆö the agent can't see what options exist without actually opening the dropdown).
- **recommendation**: adopt (select-options preview + date format hint only; skip range/file/audio/video compound synthesis as lower value for a keyless agent that can just try `type`/`fill` and observe the result)
- **change**: In walk.ts's `visit()`, when `nodeName === 'select'`, read `dom` node's DOM children (already available via `doc.root` walked in `collectDom` ŌĆö extend `collectDom` to also record `<option>` text/value under each `<select>`'s backendNodeId) and append an `options` synthetic field to `SnapNode` (e.g. `optionsPreview: string[]`), rendered by serialize.ts's `formatLine` as `options=A|B|C|... (N more)`. Separately, when `attrs['type']` is one of `date|time|datetime-local|month|week`, set `flags.placeholder` to the ISO format string if no explicit placeholder attribute exists (mirrors browser-use's format_map, lines 1116-1121).
- **keyless_ok**: true
- **priority**: P1
- **evidence**: browser-use `dom/serializer/serializer.py:271-298,335-415` (select options), `:1110-1140` (date/time format map); moxxie `perception/walk.ts:229` (placeholder is the only hint field), `perception/roles.ts` (no options/format concept exists).

### 4. [P2, align] Cursor-interactive scan has no `has_js_click_listener` (framework onClick) detection
- **source_does**: `clickable_elements.py::is_interactive` treats `node.has_js_click_listener` (populated upstream via CDP `DOMDebugger.getEventListeners` or similar) as an immediate `return True` ŌĆö this is explicitly called out as the mechanism that catches Vue `@click`, React `onClick`, Angular `(click)` handlers that are attached via framework event delegation rather than a literal `onclick=` DOM attribute or inline handler.
- **moxxie_current**: SCAN_JS's clickability heuristic (walk.ts:396-430) checks `el.hasAttribute('onclick') || el.onclick !== null`, `style.cursor === 'pointer'`, `tabindex`, and `contenteditable` ŌĆö it has no equivalent of querying actual attached event listeners. React/Vue apps typically attach listeners via addEventListener at a root/delegated level, which `el.onclick` won't see, so moxxie is relying entirely on the `cursor:pointer` CSS heuristic to catch these ŌĆö which works for well-styled buttons but silently misses div/span "clickables" that don't bother setting `cursor:pointer` (rare but real, e.g. drag handles, custom checkboxes styled without a pointer cursor).
- **recommendation**: align (cheap, no need for full CDP `DOMDebugger.getEventListeners` round trip per element ŌĆö that would be slow for SCAN_ELEMENT_LIMIT=10k elements)
- **change**: This is not worth adopting as browser-use does it (a per-node CDP round trip is too slow keyless at scale). Skip the literal mechanism; document as accepted gap. If desired, a low-cost partial mitigation: SCAN_JS could additionally treat `getEventListeners`-style detection via a single one-shot `Object.getOwnPropertyDescriptor`-based heuristic ŌĆö but this is genuinely hard without CDP's runtime introspection. Recommend leaving as-is; not high value relative to cost.
- **keyless_ok**: true (both current + declining to change)
- **priority**: P2
- **evidence**: browser-use `dom/serializer/clickable_elements.py:38-42` (`has_js_click_listener`); moxxie `perception/walk.ts:403` (`hasOnClick` = attribute/`el.onclick` only).

### 5. [P1, skip-cargo-cult] Full rect-union paint-order algorithm (`RectUnionPure`, up-to-5000-rect fragmentation)
- **source_does**: `paint_order.py` implements a general polygon-clipping rectangle-union structure (`_split_diff`, `contains`, `_MAX_RECTS = 5000` safety cap against exponential fragmentation) to compute exact occlusion across arbitrarily many overlapping layers, ordered by DOM paint order.
- **moxxie_current**: absent (see finding #2).
- **recommendation**: skip-cargo-cult (as the general mechanism) ŌĆö adopt only the narrow single-point `elementFromPoint` heuristic in finding #2. The full rect-union is solving a harder problem (exact coverage across N stacked layers) than moxxie needs; moxxie's action layer (`actuation/actions.ts:293` already calls `scrollIntoViewIfNeeded` before acting) and Playwright's own actionability checks (`click()` will throw if the target is covered) already provide a second line of defense at act-time. moxxie only needs perception-time occlusion to avoid *offering* obviously-dead refs, not to perfectly replicate paint order ŌĆö a single-point hit test is 90% of the value at ~1% of the implementation cost.
- **keyless_ok**: true
- **priority**: P1 (as a documented "don't build this" ŌĆö pairs with finding #2's cheaper version)
- **evidence**: browser-use `dom/serializer/paint_order.py:1-60` (RectUnionPure); moxxie `actuation/actions.ts:293` (`scrollIntoViewIfNeeded`) already provides act-time affordance-checking that overlaps with what full paint-order filtering buys.

### 6. [P2, adopt] Duplicate-attribute-value dedup + attribute-vs-name collapse not done in moxxie's line formatting
- **source_does**: `_build_attributes_string` (serializer.py:1230-1276) drops attributes whose value duplicates another already-shown attribute (e.g. `aria-label` == visible text, `title` == `aria-label`), protecting a small allowlist (`format`, `placeholder`, `value`, ...) from removal, and drops `role` if it equals the tag name, drops `type` if it equals the tag name, drops `aria-expanded` in favor of the AX `expanded` property, and drops `invalid='false'`/`required='false'` (only shows attributes when true/meaningful).
- **moxxie_current**: `formatLine` in serialize.ts (walk.ts consumer) always emits every non-empty flag (`checked`, `expanded`, `selected`, `disabled`, `required`, `placeholder`, `url`) with no dedup against `name` ŌĆö e.g. if `aria-label` and visible text are identical, moxxie's `accessibleName` resolution (accessible-name.ts, not read in depth here but referenced by walk.ts:207-213) already picks ONE source per the accessible-name algorithm, so this exact duplication class is less likely to occur in moxxie's model (single `name` field, not multiple raw attributes echoed) ŌĆö this is architecturally different enough that literal duplicate-stripping doesn't directly transfer.
- **recommendation**: skip-cargo-cult ŌĆö moxxie's single resolved `name` field (via `accessibleName()`) already avoids the multi-attribute duplication problem browser-use is solving; re-introducing raw attribute echoing + a dedup pass would be strictly worse (more surface, re-adding a problem moxxie's design sidesteps).
- **keyless_ok**: true
- **priority**: P2
- **evidence**: browser-use `serializer.py:1230-1276`; moxxie `perception/walk.ts:206-213` (`accessibleName()` already resolves to one name from `ariaLabel`/`alt`/`title`/`placeholder`/`textContent` ŌĆö no raw multi-attribute echo to dedup).

### 7. [P1, adopt] Scrollable-container-as-interactive heuristic for dropdown/menu/listbox containers without interactive descendants
- **source_does**: `serializer.py::_assign_interactive_indices_and_mark_new_nodes` (lines 672-707) specifically indexes scrollable containers as interactive targets when they look like a dropdown (`role in listbox/menu/combobox/menubar/tree/grid`, tag `select`, or class name matching `dropdown`/`select-menu` patterns) OR ŌĆö for any other scrollable element ŌĆö only if it has *no* interactive descendants (avoiding double-indexing a scrollable `<div>` that merely wraps buttons).
- **moxxie_current**: `walk.ts` has no scrollability concept at all ŌĆö `SnapNode` carries no `isScrollable`/`isActuallyScrollable` field, and `refEligible` is purely `isInteractive || (isContent && named) || cursorInteractive` (walk.ts:236). A scrollable custom dropdown/listbox container that AX marks `generic` (not a native role) and that has no cursor-interactive signal (no `cursor:pointer`, no `onclick`, no `tabindex`) is invisible to the agent as a distinct "scroll this to reveal more" target ŌĆö the agent has to guess it needs `mouse.wheel` over some coordinate rather than being told "this ref is a scrollable region."
- **recommendation**: adopt
- **change**: Add a cheap scrollability check to SCAN_JS (`el.scrollHeight > el.clientHeight + 1 || el.scrollWidth > el.clientWidth + 1` combined with `overflow`/`overflow-y` computed style being `auto|scroll`), thread a `isScrollable` bool through `ScanRecord`/`CursorInfo` into `SnapNode`, and in walk.ts's `visit()` OR it into `refEligible` when the node has no interactive descendants already covered elsewhere in the tree ŌĆö cheaper than browser-use's `_has_interactive_descendants` full subtree walk since moxxie's DFS is already single-pass top-down; a simpler approximation (skip the "no interactive descendants" check, just always flag scrollable custom-dropdown-shaped containers by class/role match) is acceptable as a first pass.
- **keyless_ok**: true
- **priority**: P1
- **evidence**: browser-use `dom/serializer/serializer.py:672-707`; moxxie `perception/walk.ts` (`SnapNode` type, walk.ts:33-56) has no scrollability field anywhere.

### 8. [P2, align] iframe "hidden content below" hint text is absent, and moxxie has no scroll-affordance narration at all
- **source_does**: `serialize_tree` (serializer.py:1068-1084) appends `... (N more elements below viewport - scroll to reveal)` lines listing specific hidden interactive elements with an estimated scroll distance in "pages", for both generic scrollable content and specifically for iframe content beyond a viewport threshold.
- **moxxie_current**: No such narration exists ŌĆö moxxie's snapshot is DOM/AX-based (not viewport-clipped), so in principle all *visible-in-the-page-layout* interactive elements are already included regardless of current scroll position (this is a different, arguably better, design choice ŌĆö the agent doesn't need to be told to scroll to "reveal" elements the DOM already contains, assuming they're laid out and not `display:none`). This finding is therefore mostly moot for moxxie's architecture.
- **recommendation**: skip-cargo-cult ŌĆö moxxie's non-viewport-clipped capture already solves the underlying problem (agent missing off-screen elements) more directly than a "scroll hint" narration would; adopting the hint text would be solving a problem moxxie's design doesn't have. The one adjacent thing worth keeping in mind: if moxxie ever adds viewport-only compact modes for token budget reasons, this hint pattern would become relevant then ŌĆö not now.
- **keyless_ok**: true
- **priority**: P2
- **evidence**: browser-use `serializer.py:1068-1084`; moxxie `perception/walk.ts` (AX walk is document-wide via `Accessibility.getFullAXTree`, not viewport-scoped ŌĆö confirmed no `getBoundingClientRect` viewport-intersection gate anywhere in the AX DFS, only in the SCAN_JS cursor-candidate rect check at walk.ts:413-414 which only requires non-zero size, not on-screen).

## Top recommendation
Fix the hardcoded `frameId: 'main'` / missing iframe-AX-walk gap (finding #1) first ŌĆö it's a correctness hole (silently invisible interactive content inside any same-origin iframe: payment forms, embedded checkouts, consent widgets, many SaaS admin panels that embed sub-apps) that's cheap to close with existing CDP primitives moxxie already uses (`Accessibility.getFullAXTree` supports a `frameId` param; `page.frames()` gives Playwright's frame tree for free), and it's the kind of gap that fails silently ŌĆö the agent just doesn't see the form, with no error signal.
