# Top 5 things browser-use does better than Silver

Read directly from `/Users/seventyleven/Desktop/Silver/reference/browser-use/browser_use/` (real code, not the r2 digests, though those were cross-checked). Silver comparison points read from `/Users/seventyleven/Desktop/Silver/silver/src/`.

## 1. DOM serializer: full render-tree cascade, not just an AX-tree walk

`dom/serializer/serializer.py` (1290 lines) builds the LLM-facing tree through a five-stage pipeline: `_create_simplified_tree` (shadow-DOM/iframe recursion, SVG-child pruning at `serializer.py:20-38`) â†’ `PaintOrderRemover` (occlusion) â†’ `_optimize_tree` â†’ `_apply_bounding_box_filtering` (`serializer.py:729-838`: drops a child â‰Ą99%-contained in a propagating parent like `<a>`/`<button role=combobox>`, with five exception rules â€” form elements, onclick, aria-label, interactive role, nested propagator) â†’ `_assign_interactive_indices_and_mark_new_nodes`. It also synthesizes virtual "compound components" for native controls (`_add_compound_components`, `serializer.py:150-333`): a `<input type=range>` gets a synthesized `slider` node with min/max; `<select>` gets its first-4-options list plus a `format_hint` (numeric/date/email) inferred from option values (`_extract_select_options`, line 335); date/time inputs get an ISO-format hint injected as an attribute (`_build_attributes_string`, line 1102-1177) specifically to stop LLMs guessing locale-formatted dates.

**GAP: Silver should adopt this.** `src/perception/serialize.ts` (292 lines) is a flat AX-tree-to-text renderer (`walk.ts` produces `SnapNode[]`, `serialize.ts` just formats + mints refs + redacts + compacts). It has no bounding-box occlusion filtering, no compound-component synthesis, and no date-format hinting â€” a `<select>` or `<input type=date>` renders as a bare role/name/value line with none of the disambiguating context browser-use gives the model. Silver's format IS more token-efficient per-node (plain-text lines vs. attribute-heavy pseudo-HTML), which is a real, opposite-direction win worth keeping â€” but the compound-component/date-hint layer is a pure quality-of-model-output win Silver is leaving on the table, and it's cheap to port since Silver already has node metadata (`walk.ts`) to hang it off of.

## 2. Action registry: decorator-driven dynamic action set with auto-synthesized Pydantic param models

`tools/registry/service.py`'s `Registry.action(...)` decorator (line 291) is the extensibility spine of browser-use: any `async def foo(params: FooParams, browser_session: BrowserSession, ...)` becomes a callable LLM tool by decoration alone. `_normalize_action_function_signature` (line 79) inspects the function signature, separates "special" injected params (`browser_session`, `file_system`, `has_sensitive_data`, etc., enumerated in `_get_special_param_types`, line 62) from action params, and auto-generates a Pydantic model via `create_model` when none is supplied (line 150) â€” so a plugin author writes a plain typed function and gets validation, domain-scoping (`domains=`), and a `terminates_sequence` flag for free. This is the mechanism third-party MCP/tool authors extend.

**GAP: Silver has no equivalent â€” deliberately.** `src/security/registry.ts` is a *phase-quarantine* registry: a hardcoded `Set<string>` of verb names (`READ_ONLY_VERBS`, `ACTOR_VERBS`) gated by `--enable-actions`/`readOnly`, and `src/actuation/actions.ts` dispatches through a fixed `ActVerb` union via switch/case, not a decorator-registered dynamic action set. This is intentional â€” Silver's whole security posture (quarantine-as-code, spec Â§7) depends on the dispatchable set being a closed, statically-auditable list, which is the opposite goal of browser-use's open extensibility. Silver should NOT copy the dynamic-registration mechanism (it would reopen the exact prompt-injection surface the quarantine registry exists to close) â€” but the *auto-generated param validation* idea (Pydantic `create_model` from a function signature) is a reusable pattern if Silver ever wants schema-validated verb args without hand-writing a Zod schema per verb.

## 3. `multi_act` page-change guard: two-layer, mid-batch abort with partial-results preservation

`agent/service.py:2719-2838`. Before executing a queued batch of actions from one LLM turn, browser-use snapshots `pre_action_url`/`pre_action_focus`; after each action it checks a **static** guard (`registered_action.terminates_sequence` â€” declared on actions like `navigate`/`search`/`go_back`, line 2806) and a **runtime** guard (URL or focused target actually changed, line 2812-2818) â€” either one aborts the remaining queue immediately, logging how many actions were skipped and why. On any exception it appends a partial `ActionResult(error=...)` and returns what succeeded so far (line 2820-2836) rather than losing state. `done` is also only allowed as the sole action if not first in a batch (line 2751-2755).

**Silver has this, functionally equivalent, different mechanism.** `src/actuation/pagechange.ts` computes a settle-then-fingerprint (`url + focusedBackendNodeId + domNodeCount`, `pagechange.ts:34,71-83`) and stamps `page_changed`/`stale_refs` as a **flag** on the response rather than auto-aborting a queue â€” this is a deliberate design choice documented at the top of the file ("the CLI wants to tell the host... it must NEVER auto-embed a fresh snapshot... that is the host's decision"). Since Silver dispatches one verb per CLI invocation (the host LLM decides the next action after seeing the flag) rather than executing a pre-planned multi-action batch per turn, there is no "queue" to abort mid-flight â€” the equivalent safety property (don't act on stale refs) is enforced per-call by `groundRef`'s generation check in `actions.ts:134-136`, which fails BEFORE any Playwright call on a stale/unknown ref. `src/cli.ts`'s `batch` verb (line 2286+) does chain multiple CLI commands in one process, but per the code it holds no cross-command abort-on-page-change logic beyond `--bail` on error â€” Silver's batch mode could adopt browser-use's mid-batch page-change abort (stop remaining batched commands once a page-changing one succeeds) rather than relying solely on the next command's own ref-generation check to fail loudly. Marking this a partial gap: the safety invariant is provably equivalent, but batch-mode UX (skip-with-reason vs. fail-on-next-command) is worse in Silver.

## 4. Watchdog decomposition: 13 single-responsibility event-driven services vs. one monolithic path

`browser/watchdogs/` has 13 files (`dom_watchdog.py`, `crash_watchdog.py`, `downloads_watchdog.py`, `popups_watchdog.py`, `security_watchdog.py`, `permissions_watchdog.py`, `captcha_watchdog.py`, `storage_state_watchdog.py`, `har_recording_watchdog.py`, `recording_watchdog.py`, `screenshot_watchdog.py`, `aboutblank_watchdog.py`, `local_browser_watchdog.py`), each subscribing to specific `bubus` events (`LISTENS_TO`/`EMITS` class attrs, e.g. `DOMWatchdog.LISTENS_TO = [TabCreatedEvent, BrowserStateRequestEvent]`, `dom_watchdog.py:34`). `DOMWatchdog` alone shows the value: it races DOM-tree build against screenshot capture in parallel (`_build_dom_tree_without_highlights` / `_capture_clean_screenshot` via `create_task_with_error_handling`, lines 356-385), does a bounded 2s pending-network-request probe with graceful timeout fallback (line 267-279), and caches a `DomService` instance across calls to reuse the CDP client connection (line 546-554, `__del__` explicitly avoids tearing it down).

**GAP: partially â€” this is architecture-scale, worth adopting selectively.** Silver has no watchdog/event-bus layer; DOM building, screenshot, and popup/permission handling are inline per-verb logic in `handlers.ts`/`actions.ts`, reconnecting CDP per command (per the task brief's stated engine-efficiency question). The concrete, portable win here is narrower than "adopt an event bus": it's (a) parallel DOM+screenshot capture instead of serial, and (b) reusing one CDP session/DomService instance across a command sequence instead of reconnecting per verb â€” both directly attack the "per-command CDP reconnect" latency gap named in the task brief. Silver's `core/session.ts` should be checked for whether it already pools a CDP connection (out of scope for this pass; flagging as the highest-leverage adoption from this file, not the full watchdog decomposition, which is more machinery than Silver's single-process CLI needs).

## 5. Sensitive-data handling: real value never enters the LLM's context, only a `<secret>key</secret>` placeholder

`tools/registry/service.py:427-490` `_replace_sensitive_data` and `tools/service.py:781-830`. The agent is constructed with a `sensitive_data` dict the LLM never sees the values of; the LLM emits `<secret>login_pw</secret>` in an action's text param, and only at execution time â€” after Pydantic validation, right before the Playwright call â€” does `_replace_sensitive_data` swap the placeholder for the real value via regex (`secret_pattern = re.compile(r'<secret>(.*?)</secret>')`, line 442). Values are domain-scoped (new dict format `{domain_pattern: {key: value}}`, matched via `match_url_with_domain_pattern` against the *current URL*, line 452-459 â€” a secret for `bank.com` is inapplicable on `evil.com` even if the LLM is prompt-injected into requesting it), and the special-param plumbing in `execute_action` (`service.py:377-379`) restricts `sensitive_data` to only the `input` action, not the whole special-context dict every action receives.

**GAP: Silver should adopt this â€” it's the largest security-relevant gap found.** Silver's `src/security/redact.ts` only protects the READ path: `redactValue` masks password-shaped/card-shaped values so they never appear back in a *snapshot* (`redact.ts:44-54`, confirmed by grep â€” no `<secret>`/`vault`/`fromEnv` handling anywhere in `cli.ts` or `actuation/actions.ts`). There is no WRITE-path equivalent: a `silver fill` targeting a password field takes the literal secret value as a CLI argument, meaning the value must already be visible to whatever produced that command line (the host LLM's own context, or an env var it read and echoed). Browser-use's `<secret>` indirection means the LLM can drive a login flow â€” "type the saved password into ref=e7" â€” without the raw credential ever being tokenized into the model's context window at all. Given the task brief's premise that "Host LLM is the brain" and Silver is explicitly keyless, this is the one item that's in tension with Silver's own threat model: keyless-for-the-model-provider is not the same guarantee as secret-never-in-context, and right now Silver has the former but not the latter. Concretely: add an `--secret name=value` (or `SILVER_SECRET_<name>` env-var) registration step, resolve `<secret>name</secret>` inside `fill`/`type` values at the actuation choke point (mirroring where `redactValue` already sits in the READ path), and domain-scope it against the current page URL the same way browser-use does.
