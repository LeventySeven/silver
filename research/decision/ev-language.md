# ev-language: Rust vs TypeScript vs Python for the Silver browser CLI

Scope: objective fact digest, grounded in the three real codebases on disk —
`silver` (Rust fork of Vercel agent-browser), `skill/agent-browser` (our TS/Playwright CLI, codename "moxxie"),
`reference/webwright` (Microsoft, Python/Playwright). All claims below are anchored to files I read directly.

## Facts

### Distribution / install-and-use
- **silver (Rust)**: ships as a prebuilt native binary per platform. `bin/silver.js` is an npm shim; root `package.json` scripts cross-compile to `bin/agent-browser-darwin-arm64`, `-darwin-x64`, plus Linux/Windows via Docker (`build:linux`, `build:windows` in `/Users/seventyleven/Desktop/Silver/silver/package.json`). README (`/Users/seventyleven/Desktop/Silver/silver/README.md:9-42`) documents four install paths: `npm install -g agent-browser`, Homebrew, `cargo install agent-browser`, or build from source. `postinstall: node scripts/postinstall.js` fetches the matching platform binary. Chrome itself is fetched separately: `agent-browser install` downloads Chrome for Testing into `~/.silver/browsers` (`cli/src/install.rs:9-14`, `get_browsers_dir`).
- **TS (moxxie)**: pure npm package, `bin: {"moxxie": "./dist/cli.js"}` (`/Users/seventyleven/Desktop/Silver/skill/agent-browser/package.json`). `dependencies: {"playwright": "^1.61.0"}` — Chromium download is Playwright's own postinstall, a well-trodden npm path. No native compile step; `engines.node >=24`.
- **webwright (Python)**: `pyproject.toml` declares `webwright = "webwright.run.cli:app"` entry point, installed via pip/uv; `dependencies` include `playwright>=1.45`, `typer`, `pydantic`, `httpx`, `rich`. Playwright-Python also requires a separate `playwright install` browser-download step (standard for any Playwright-Python project, mirrors Chrome for Testing pattern).
- Net: Rust is the only one that ships as a truly dependency-free single static binary once installed (no runtime interpreter needed to execute the CLI itself); TS and Python both require their language runtime (Node ≥24, Python ≥3.10) to be present, but both piggyback on Playwright's mature browser-provisioning path rather than needing a bespoke CDP-download shim like silver's `install.rs`.

### Startup + runtime speed
- Not benchmarked directly in the codebases I read (no perf numbers found in `silver/evals` or `skill/agent-browser`), so I make no numeric claim. Structurally: both silver and moxxie implement a **daemon / persistent-browser-process model** specifically to amortize startup cost across CLI invocations:
  - silver: `cli/src/connection.rs` implements a Unix-socket daemon (`walk_daemons`, `ensure_daemon`, `daemon_ready`, `EnsureDaemonResult`) that a new CLI invocation reconnects to rather than relaunching Chrome each call.
  - moxxie: `src/core/session.ts:2-4` — comment reads "the load-bearing 'browser-as-daemon' model (plan Task 4)" — `openSession` spawns a **detached** Playwright-launched Chromium (`spawn(execPath, args, {detached: true})` at line 147) and later CLI calls reconnect over CDP (`session.ts:237-246`).
  - webwright has no equivalent daemon/detach code found in `environments/local_browser.py`; it drives Playwright via `asyncio` inside its own process lifetime (`async_playwright().start()` at line 316), i.e., closer to a single-run script model, consistent with its "task = re-runnable script" design goal.
- silver's `cli/Cargo.toml` release profile is tuned for a small/fast binary (`opt-level = 3, lto = true, codegen-units = 1, strip = true`), typical Rust CLI practice — evidence of intent to optimize binary size/startup, not proof of a specific number.

### Browser-control ecosystem
- **silver (Rust)**: hand-rolled CDP. `cli/Cargo.toml` dependencies include `tokio-tungstenite` (raw WebSocket client) and no Playwright/chromiumoxide dependency anywhere in the manifest. `cli/cdp-protocol/` contains raw `browser_protocol.json` and `js_protocol.json` (the CDP spec files), which the CLI presumably codegens/parses against directly. This confirms the task brief's premise: Vercel's agent-browser talks CDP itself rather than going through a browser-automation library.
- **TS (moxxie)**: `playwright` is a first-class direct dependency (`"playwright": "^1.61.0"`), imported directly in `src/core/session.ts`. Playwright is TS-native — written and maintained by Microsoft in TypeScript, with the Node API as the primary/reference API.
- **webwright (Python)**: `playwright>=1.45` dependency, uses `playwright.async_api.async_playwright` (`environments/local_browser.py:311,316`). Playwright's Python bindings are official (Microsoft-maintained) but are a generated/wrapped port of the Node implementation, not the primary implementation surface.
- Practical implication: silver's Rust CLI owns 100% of protocol-compatibility burden itself (must track every CDP domain/command by hand against the JSON protocol files); moxxie and webwright both inherit Playwright's auto-wait, selector engine, network interception, and multi-browser (Chromium/Firefox/WebKit) support for free.

### Dev velocity / enhanceability for us
- Silver's Rust surface is large and manual: `cli/src/` has 15+ top-level modules (`chat`, `commands`, `connection`, `doctor`, `flags`, `install`, `mcp`, `native`, `output`, `plugins`, `read`, `skills`, `upgrade`, `validation`) — every new Chrome feature/flag requires touching Rust code and the CDP protocol layer directly (evidenced by `native/actions.rs`, `native/egress.rs` housing the SSRF guard logic).
- moxxie's src is organized by concern that maps onto agent needs: `core/`, `security/`, `extract/`, `perception/`, `actuation/` — `security/` alone has 5 files (`confirm.ts`, `egress.ts`, `injection.ts`, `redact.ts`, `registry.ts`), i.e., the safety surface the task brief credits to "our" TS work is real and present in the tree.
- moxxie has 14 test files totaling 142 individual `it(`/`test(` cases (counted directly: `tests/unit/security.test.ts`=57, `errors.test.ts`=7, `serialize.test.ts`=18, `smoke.test.ts`=2, `diff.test.ts`=6, `extract.test.ts`=22, `roles.test.ts`=5, `refmap.test.ts`=9, `integration/iframe.test.ts`=2, `walk.test.ts`=1, `security-harden.test.ts`=10, `actions.test.ts`=1, `session.test.ts`=1, `cli.test.ts`=1 — sums to 142), confirming the "142 tests" figure in the task brief is accurate as a case count, not file count.
- I could not find a `pass_k`/`pass@1` eval number in either `skill/agent-browser` or `silver/evals` to verify the "eval pass_k 1.000" claim — silver does have a separate `evals/` harness (`run.ts`, `lib/judge.ts`, `cases/skill-selection.ts` etc.) but it evaluates skill-selection/context-footprint, not task-completion pass@k, and I found no matching artifact for moxxie's claimed number. Treat that specific figure as unverified from the files I read.
- Rust's compile-check discipline (ownership, exhaustive `Result`/`Option`) generally slows iteration speed per change vs. TS/Python but catches a class of bugs earlier; this is a structural tradeoff, not something I can quantify from the repo alone.

### Concurrency model for parallel browsers
- silver: `tokio` with `rt-multi-thread` feature (`cli/Cargo.toml`), 70 call sites of `tokio::spawn`/`tokio::task` inside `cli/src` — genuine multi-threaded async runtime, well-suited to driving many CDP WebSocket connections concurrently with true OS-thread parallelism where needed.
- moxxie (Node): single-threaded event loop; concurrency achieved via `Promise`-based async I/O and OS-process-level parallelism through `child_process.spawn` (`src/core/session.ts:14`, `src/cli.ts:46` — "tests invoke directly (no child_process)"), i.e., each parallel browser is a separate detached Chromium process, and JS-side orchestration across them is cooperative (event-loop), not preemptive.
- webwright (Python): `asyncio` event loop (`environments/local_browser.py:3,219,253,437`), same cooperative-concurrency shape as Node — single-threaded event loop with an explicit `_ensure_loop()`/`asyncio.new_event_loop()` pattern, plus the classic Python GIL constraint if any CPU-bound work were added (none observed in the browser driver itself, which is I/O-bound).
- For the stated use case (many parallel browser sessions doing I/O-bound CDP traffic), tokio's multi-threaded executor and Node/Python's single-threaded event loops are all viable — the difference shows up only if the host process also wants CPU-bound parallel work (e.g., parallel extraction/parsing), where tokio has a structural edge and Python's GIL a structural disadvantage.

## Pros / Cons by criterion

| Criterion | Rust (silver) | TypeScript (moxxie) | Python (webwright) |
|---|---|---|---|
| (a) Install/distribution | + single static binary, no runtime needed; − must cross-compile+ship binaries per OS/arch (`build:linux/windows/macos` scripts), plus bespoke Chrome-download logic (`install.rs`) | + plain npm install, Playwright owns browser provisioning; − needs Node ≥24 runtime present | + pip/uv install, Playwright owns browser provisioning; − needs Python ≥3.10 runtime + separate `playwright install` step |
| (b) Startup/runtime speed | + no interpreter startup, LTO-tuned release binary; both silver and moxxie mitigate cold-start via a daemon/detached-process model so steady-state startup cost is engineering-choice-driven, not language-driven | daemon-mitigated per `session.ts`; Node cold start still slower than a native binary in principle (unverified numerically here) | no daemon/detach code found; likely relaunches Playwright/browser each script run, consistent with its "one-shot resumable script" design, not "fast quick task" design |
| (c) Browser-control ecosystem | − hand-rolled CDP client against raw `browser_protocol.json`/`js_protocol.json`, all auto-wait/selector/network-intercept logic must be built and maintained in-house | + Playwright is TS-native, richest/most current API surface, direct dependency in `package.json` | + Playwright-Python is official but a secondary/generated binding surface relative to the Node implementation |
| (d) Dev velocity/enhanceability for us | − 15+ manual modules, must extend CDP coverage by hand; compiler rigor slows iteration | + existing modular structure (`security/`, `extract/`, `perception/`, `actuation/`) already houses the security/extract work credited to "us"; 142 passing test cases give a real regression net | moderate — dynamic typing speeds prototyping but no test/module evidence found comparable to moxxie's structure |
| (e) Concurrency for parallel browsers | + multi-threaded tokio runtime (70 spawn/task sites), true parallelism available | cooperative event loop + OS-process-level parallel Chromium instances via `child_process.spawn` | cooperative asyncio event loop; GIL is a structural ceiling if CPU-bound work is ever added to the driver |

## Caveats / things I could not verify from the repos
- No numeric startup-time or throughput benchmarks exist in any of the three trees for a head-to-head speed claim — the daemon architecture in both silver and moxxing is the load-bearing evidence for "we've engineered around cold start," not a stopwatch number.
- The "eval pass_k 1.000" figure attributed to moxxie could not be located in `skill/agent-browser` or `silver/evals`; the 142-test-case figure was independently verified by direct count and matches.
- I did not execute either binary/CLI to confirm they run cleanly (out of scope for this fact digest; static grounding only).
