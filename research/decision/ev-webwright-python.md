# Evidence: Webwright (Microsoft, Python + Playwright)

## Facts

### What Webwright actually is
- Package `webwright`, `pyproject.toml:6-20`: deps are `httpx, jinja2, pydantic, pyyaml, rich, typer, playwright, python-dotenv, platformdirs`. ~1.5k LoC total per README (`README.md:13,54`).
- It is an **LLM-driven agent harness that calls a model itself** (OpenAI/Anthropic/OpenRouter backends, `src/webwright/models/{openai,anthropic,openrouter}_model.py`). It is NOT keyless in its native form — every `python -m webwright.run.cli` invocation makes real model API calls (`README.md:174,185-197`, `src/webwright/models/base.py`, `config/model_openai.yaml`/`model_claude.yaml`).
- The **only keyless path** is the Claude-Code/Codex/OpenClaw/Hermes "skill" adaptation (`skills/webwright/SKILL.md:9-21`), which explicitly replaces the internal LLM loop: "you are the Webwright agent... you replace that loop directly... No `OPENAI_API_KEY` or other model API keys required" (`SKILL.md:9-21`). In that mode the *host* agent (e.g. Claude Code) IS the brain and issues `Bash` calls itself, following the workspace contract as a set of instructions/prompts, not as Python code that runs.

### Long-running-task model — "task = re-runnable script, logs = artifact"
- Core loop is `DefaultAgent` in `src/webwright/agents/default.py`. Each `step()` = one model `query()` → one JSON action (`bash_command` or `python_code`) → `env.execute()` → observation appended to `messages` (`default.py:383-436`). This is a **single-session, single-process while-loop** (`default.py:363-381`), not a job scheduler; there is no separate "task" object beyond the in-memory `messages` list and a `trajectory.json` dump after every step (`default.py:371,438-467`).
- The "re-runnable script" claim is a **workspace convention enforced by the system prompt**, not a language feature: `base.yaml:78-116,289-315` mandates the agent create `final_runs/run_<id>/final_script.py`, re-run it from scratch on each fix, and gate `done=true` behind a self-reflection judge reading `final_runs/run_<id>/self_reflect_result.json` with `predicted_label==1` (`base.yaml:289-315`, enforced in code by `DefaultAgent._tool_gate_error` / `_self_reflection_gate_error`, `default.py:202-268,402-424`). If the judge file is missing or `predicted_label != 1`, `execute_actions` force-flips `extra["done"]=False` and re-injects an error message (`default.py:403-412`) — this gate IS real code, not just prompt text.
- "Logs as artifact": every step is persisted to `steps_dir`/`logs_dir` in `local_workspace.py` (`local_workspace.py:133-151`) or `local_browser.py` (`_persist_step_code`, `local_browser.py:422-430`), plus a running `trajectory.json` (agent-level) and, in local-browser mode, per-step screenshots (`local_browser.py:463-502`). Debug markdown (`debug/steps.md`) is also written per step when `debug_log: true` (`default.py:93-176`).
- **Context compaction for long horizons**: `summary_every_n_steps` (default 20 in `base.yaml:90`) triggers `_compact_history()` which makes one more LLM call to summarize the transcript and collapses `self.messages` down to `[system, summary]` (`default.py:303-339`). Also `keep_last_n_observations` prunes ARIA snapshots from older observation messages to bound token growth (`default.py:275-301`, enabled in `local_browser.yaml:85`). Step budget is hard-capped by `step_limit` (default 100 in `base.yaml:87`, 15 in `AgentConfig` default) and enforced via `LimitsExceeded` (`default.py:386-394`).
- **Persistent browser across steps despite a stateless script model**: `tools/persistent_local_browser.py` spawns a detached headless Chromium subprocess with a fixed CDP port, writes `{id,pid,connectUrl,userDataDir}` to `.lb_session.json`, and every subsequent bash step attaches via `connect_over_cdp` then calls `browser.close()` (CDP-only close — the Chromium process survives) — this is how "one action per bash command" combines with cookie/DOM state persisting across turns (`persistent_local_browser.py:1-25`, `persistent_browser.yaml:1-16,105-172`). Release is a required step before `done=true` (`persistent_browser.yaml:111,317-321,443-451`) or it leaks a zombie process — this is an explicit lifecycle bug users must self-police, not automatic.
- There is also a fully separate **live-browser mode** (`local_browser.yaml`) where the model drives `page/context/browser` directly with free Python each step and there is NO workspace/final-script contract at all (`local_browser.yaml:9-13,65-133`) — this is the "quick task" mode, architecturally distinct from the "long task = script" mode.

### Perception
- In workspace/script mode: no vision by default. The agent gets text observations (URL, title, stdout, ARIA snapshot text) via `LocalWorkspaceEnvironmentConfig`/templates (`local_workspace.py:230-273`, `base.yaml:26-39`). It must explicitly call `webwright.tools.image_qa` (a separate LLM vision call) to "see" screenshots (`base.yaml:184-190,206-286`). `model.attach_observation_screenshot: false` by default (`base.yaml:26`, `local_browser.yaml:22`).
- In live-browser mode: same — ARIA snapshot of `body` is the primary signal every step (`local_browser.py:479-484`, `local_browser.yaml:15-19,48-49`); screenshot path is given as text only, not attached as an image, unless explicitly overridden (`local_browser.yaml:20-29`).
- `image_qa` and `self_reflection` (`src/webwright/tools/image_qa.py`, `self_reflection.py`) are themselves LLM-calling tools (vision-model judge), i.e. Webwright natively is not just "one brain" — it spawns extra model calls for perception/verification. The skill adaptation explicitly strips these and substitutes the host agent's own multimodal `Read` on PNGs (`SKILL.md:19-21,98-105`).

### Skill form
- Shipped as `skills/webwright/SKILL.md` + `commands/{run.md,craft.md}` + `reference/{playwright_patterns.md, cli_tool_mode.md, workflow.md}` (file listing above). Designed to be symlinked/loaded across four hosts (Claude Code, Codex, OpenClaw, Hermes) — same folder, host-specific plugin manifests are thin wrappers (`README.md:211-347`, `.claude-plugin/plugin.json`, `.codex-plugin/plugin.json`).
- Two modes exposed as slash commands: `/webwright:run` (one-shot literal-value script) and `/webwright:craft` (parameterized reusable CLI tool with argparse + Google-docstring contract, `commands/craft.md:1-62`, `reference/cli_tool_mode.md`).
- The skill is essentially a **very detailed prompt/workflow spec** re-implemented as instructions for a host LLM to follow with `Bash/Read/Write/Edit`, not code the host executes (`SKILL.md:4,71-142`). This means adopting it costs us prompt-engineering effort, not Rust/TS engineering effort — there is no library API to import.

### Multi-session / parallel support
- Grep across `src/` and `skills/` for parallel/concurrent/multi-session/multi-browser terms found **zero** hits except the phrase "parallel per-image [scoring]" inside `self_reflection`'s prompt text (`self_reflection.py:396` `asyncio.gather` — parallelizes vision-judge calls per screenshot, not browser sessions).
- `DefaultAgent.run()` is a plain synchronous `while True` loop over one `env`/one `model` instance (`default.py:341-381`); `LocalBrowserEnvironment` owns exactly one `_browser`/`_context`/`_page` (`local_browser.py:191-227`). There is no session pool, no task queue, no orchestrator for multiple concurrent agents/browsers anywhere in the codebase.
- The only way to get "N parallel agents" with Webwright as shipped is external: launch N separate OS processes (`python -m webwright.run.cli ...`), each with its own `-o outputs/<id>` and (if using CDP local-launch or Browserbase) its own browser session — this works but is entirely on the caller, not a first-class feature. Nothing in the repo manages that fan-out (no batch runner, no work-queue script). `assets/task_showcase/` is a read-only Flask dashboard for viewing completed runs, not an orchestrator (`README.md:127-165`).

### Install (pip/uv) & ease of adaptation
- Standard `pip install -e .` + `playwright install chromium` (`README.md:178-181,215-220`); no `uv`-specific instructions anywhere in README or pyproject (works under uv too since it's a normal PEP 621 project, but not documented/tested here).
- As a Rust/TS "Silver" target, the *code* (agent loop, env, models) is Python-only and would need a full rewrite to reuse in our two existing codebases — there is nothing to bind or FFI into cheaply; the ~450-line agent loop (`README.md:54`) is simple enough to reimplement logic 1:1 in either language in a day or two, but it is not a drop-in library.
- What ports over almost for free is the **workspace/prompt contract itself** (`base.yaml` system_template, `plan.md` + `final_runs/run_<id>/` + judge-gated `done`) — that's markdown/YAML prose we can copy into a Silver SKILL.md nearly verbatim, independent of language.
- `tools/persistent_local_browser.py`'s CDP-attach/detach pattern (spawn once, `connect_over_cdp` + `browser.close()`-without-killing-process per step) is a concrete, portable recipe already proven against Playwright's Python API; the identical CDP-over-websocket approach is directly reproducible in TS Playwright with no conceptual change, and in Rust it's reproducible against chromiumoxide/CDP but requires re-deriving the "close() only drops the connection, not the process" trick (Playwright-Python-specific idiom, needs re-verifying against Rust bindings). (`persistent_local_browser.py:1-25`, `persistent_browser.yaml:105-172`)

## Pros (objective, grounded)
1. Real, code-enforced completion gate for long tasks: `_tool_gate_error`/`_self_reflection_gate_error` mechanically blocks `done=true` until a judge file with `predicted_label==1` exists for the *latest* run folder (`default.py:202-268`) — stronger than a prompt-only "please verify" instruction.
2. Concrete, working persistent-local-Chromium recipe (spawn detached headless Chromium, parse DevTools ws URL from stderr, survive `browser.close()` across steps) that is directly adaptable to give Silver a genuinely long-lived, cross-step browser without keeping a whole harness process alive (`persistent_local_browser.py`).
3. Automatic context-compaction (`_compact_history`) and ARIA-snapshot pruning (`_prune_old_observation_aria_snapshots`) are real, tested mechanisms for keeping a 100-step run's token budget bounded (`default.py:275-339`) — directly useful prior art for Silver's own long-run context management, regardless of language.
4. The skill-form workspace contract (`plan.md` critical points → `final_script.py` → `final_runs/run_<id>/` → screenshot-cited self-verification) is a clean, host-agnostic re-runnable-artifact discipline that ports as pure documentation into any CLI/skill, including ours, with no code dependency.
5. Extremely small, readable codebase (~1.5k LoC, agent loop ~450 lines per README) — cheap to fully read and mine for ideas, which we just did.

## Cons (objective, grounded)
1. **Not keyless by design.** The native CLI (`webwright.run.cli`) requires an LLM API key and makes real model calls for the main loop AND for `image_qa`/`self_reflection` (`base.yaml`, `models/*`). It only becomes "keyless" when gutted down to a skill where the host model plays every one of those roles manually — meaning the actual reusable asset for Silver is the workspace-contract prompt, not the Python package.
2. **No parallel/multi-session primitive at all.** Confirmed by code grep: one `env`, one `model`, one synchronous loop per process; no task queue, no session pool, no fan-out orchestration (`default.py`, `local_browser.py`). Criterion (d) — a user launching many sub-agents sharing or each owning a browser — is entirely unaddressed; we'd build that ourselves regardless of which reference we borrow from.
3. **Zombie-process risk is manual, not automatic.** The persistent-Chromium lifecycle depends on the agent remembering to call `release` before `done=true`; the completion gate for `persistent_browser.yaml` checks for `.lb_session.json` absence (`persistent_browser.yaml:317-321,449-451`) but nothing kills orphaned Chromium processes if the agent crashes or is killed mid-run — no supervisor/reaper in code.
4. **Fragile/ad-hoc JSON action protocol.** Every model turn must emit strict single-JSON-object output (`bash_command`/`python_code`, `done`, `final_response`) parsed by `models/*_model.py`; format errors are handled via `FormatError`/`InterruptAgentFlow` re-prompting (`default.py:363-369`) — a much less structured action surface than agent-browser's discrete CLI verbs or a typed tool-call schema; harder to make robust against model output drift.
5. Python + Playwright means adopting it means running a second runtime/toolchain (Python venv, `playwright install chromium`) alongside our Rust and TS stacks — pure integration/packaging cost with no code reuse, since nothing here is exposed as a library API (only as a CLI subprocess or as prose for a skill).
6. Firefox-vs-Chromium inconsistency across their own configs (skill uses Firefox "because some sites fail under Chromium with ERR_HTTP2_PROTOCOL_ERROR", `SKILL.md:62-66`, while `persistent_local_browser.py`/`local_browser.yaml` use Chromium) — signals real-world fingerprinting friction they haven't fully resolved, something Silver would inherit either way with Playwright/Chromium.

## Relevance to Silver's 9 criteria
1. **Agent-ergonomic CLI + general SKILL.md** — Webwright's native CLI (`webwright.run.cli`) is a benchmarking harness for LLM-vs-LLM runs, not agent-ergonomic for a host LLM; its real ergonomic surface is the *skill* (`skills/webwright/SKILL.md`), which is prose-only and reusable as-is.
2. **Fast quick tasks** — covered by `local_browser.yaml`'s live-browser mode (no workspace ceremony, direct `page`/`context` driving, `keep_last_n_observations:1`) — architecturally the closest Webwright gets to agent-browser's speed, but still routed through a full JSON-per-turn LLM loop rather than direct CLI verbs.
3. **Long-running/resumable tasks** — this is Webwright's actual strength: judge-gated completion, `final_runs/run_<id>/` versioning, context compaction/pruning, persistent-Chromium-across-steps. Best mineable ideas are here.
4. **Parallel multi-agent orchestration** — unaddressed (see Con #2); nothing to adopt.
5. **Install-and-use / zero config** — requires Python env + `pip install -e .` + `playwright install chromium` + (natively) an LLM API key; the skill path removes the API-key requirement but still needs the Python/Playwright runtime installed for anything code-executing, or none at all if we only borrow the prose contract.
6. **Keyless** — native package is not keyless; only the skill-adapted, host-driven mode is, and even then perception/verification depend on the host model's own multimodal ability standing in for `image_qa`/`self_reflection`.
7. (assets already have both Vercel CLI + TS CLI) — Webwright offers no code to merge into either; only patterns (compaction, CDP-persist, judge-gate, workspace contract) worth reimplementing natively.
8/9. (single-language consolidation, adapt-the-best) — best adoptable *language-agnostic* ideas: (a) judge-gated `done` completion contract, (b) run-id-versioned artifact folders as the "long task" unit, (c) context compaction/observation-pruning strategy, (d) the CDP-attach/detach-without-killing pattern for a persistent browser across discrete tool calls.
