# ev-aside-adapt — Aside patterns worth adapting into Silver's basement

Scope: which Aside patterns (from `research/sources/aside-02/-06/-07`) should Silver adopt, keyless-compatible only, and whether the Rust base (`silver/`) or TS base (`skill/agent-browser/`) makes each easier/harder. All "current state" claims are grounded by reading the actual code, not the digests.

## Current-state facts (both bases, read directly)

- **F1.** Silver (Rust) already exposes a persistent-daemon-per-browser-session model: `silver/cli/src/main.rs:33-34` imports `ensure_daemon`, `send_command`, `walk_daemons`, `DaemonOptions` — a CLI invocation is a thin client that connects to a long-lived daemon process per session, closely matching Aside's "local daemon owns state, client is a thin console" shape (aside-02 pattern 4), but scoped to browser-driving only, not an agent-loop/model-call daemon.
- **F2.** agent-browser (TS) has the equivalent without a literal daemon binary: `skill/agent-browser/src/core/session.ts:1-12` — `openSession` spawns a **detached** Playwright-Chromium (`child.unref()`s it), survives CLI exit; cross-command state (endpoint/pid, RefMap+generation) lives in JSON sidecars under `~/.moxxie/sessions/<name>/` (`session.ts:86-99`). Comment states explicitly "NO model calls, ever."
- **F3.** Neither base has a persistent, Node/multi-statement code-execution sandbox. Both have a single-shot `eval` verb: Silver's `"eval" =>` handler at `silver/cli/src/commands.rs:987-1004` (reads one script string, optional `-b/--base64`/`--stdin`, executes once); agent-browser's `page.evaluate(...)` call at `skill/agent-browser/src/core/handlers.ts:428-430` runs one JS expression in page context per CLI invocation. Neither exposes Aside's `repl` helper surface (`openTab`, `attachBrowserTab`, `snapshot`, `fs`, `sleep`, persistent `const`/`let` across calls) — state persistence, if any, is only whatever survives inside the browser tab's own `window` globals across separate `eval` invocations, not an interpreter-level REPL scope.
- **F4.** `eval` is already flagged as a high-risk/gated verb in agent-browser's security layer: `skill/agent-browser/src/security/registry.ts:72` lists `'eval'` in the dispatchable verb set; `skill/agent-browser/src/security/confirm.ts:19-24` explicitly includes `download`/`upload`/`eval` in `MUTATING_VERBS` "by default-cautious policy (file egress/ingress and arbitrary code execution)."
- **F5.** Silver's full verb surface is 159 match arms in `commands.rs` (`grep -c '".*" =>' silver/cli/src/commands.rs` = 159), spanning navigate/click/fill/scroll/wait/screenshot/pdf/snapshot/extract/eval/auth/network/storage/cookies/tab/window/frame/dialog/trace/profiler/record/console/errors/highlight/clipboard/state/tap/mouse/etc — i.e. Silver already committed to the "N granular action tools" shape Aside's digest calls an anti-pattern (aside-02 anti-pattern 1), just exposed as CLI verbs rather than LLM tool-call schemas.
- **F6.** Neither base has any `subagent`/`subagent_wait` construct, `memory_search`/memory directory convention, `websearch`/`webfetch`-with-citation-IDs, or a skills auto-injection matcher. `grep -ril "subagent\|memory" silver/src silver/cli skill/agent-browser/src` returns no hits outside build artifacts.
- **F7.** Both bases already ship a `SKILL.md`-based skill system, but it is the **host-agent-facing** kind (one static markdown file the calling LLM reads, e.g. `silver/skill-data/core/SKILL.md`, `skill/agent-browser/SKILL.md`), not Aside's **site-specific, URL-glob-matched, auto-injected, sandboxed-library** skill engine (aside-07 patterns 14-19). `silver/skill-data/core/SKILL.md:289-291` documents multi-session parallelism ("Run multiple browsers in parallel... Each `--session <name>` is an isolated browser") but this is operator-driven CLI usage, not an agent-authored subagent-orchestration primitive.
- **F8.** Silver is Rust (`silver/cli/Cargo.toml:1-5`, edition 2021); agent-browser is TypeScript on Playwright (142 tests per project brief). Rust has no built-in dynamic-eval/sandboxed-interpreter primitive in std; a Node-level "persistent JS REPL with injected globals" (Aside's core mechanism, aside-02 pattern 1) is native to a JS/TS runtime (`node:vm`, matches Aside's own implementation — aside-07 pattern 18 confirms Aside itself runs skill bundles via `node:vm`) but requires embedding a JS engine (e.g., `boa`, `rquickjs`, `deno_core`) in Rust — extra dependency weight, not a language-native fit.

## Adaptation recommendations (grounded in F1-F8 + aside-02/-06/-07 pattern text)

### Core, adopt now
1. **Citation-ID contract for any future `webfetch`/`websearch`-equivalent tool** (aside-02 pattern 15) — cheap, additive, no existing conflicting code in either base (F6). Keyless-compatible: doesn't require a model, just a wrapper convention `source_id` + `<citation>` text. Equally easy in Rust or TS since it's just a string-formatting/ID-minting layer around whatever fetch/search backend Silver picks.
2. **Tool-registry-as-pure-function-of-session-flags** (aside-02 pattern 16) — directly maps onto Silver's existing verb dispatch. Silver's 159-verb match statement (F5) and agent-browser's `registry.ts` (F4) are both single central dispatch points already; adding a `--read-only`/`--incognito`-style flag-gated verb-set builder is a refactor of an existing chokepoint, not new architecture, in *either* language.
3. **Untrusted-tool-output wrapping for fetched/extracted page content** (aside-06 pattern 15) — agent-browser already has a `security/injection.ts` module (per file listing) suggesting this concern is partially addressed; the Aside wrap format (`[BEGIN_UNTRUSTED_TOOL_OUTPUT ...]` + neutralize forged role tags) is a straightforward string-processing addition in either base.
4. **"Mistakes & Alternatives / Avoidance" section in any session/compaction summary** (aside-06 pattern 18) — pure prompt/output-template convention, zero code-architecture dependency, trivially portable to a `SKILL.md` or session-summary format in Silver's basement regardless of language.
5. **Reading-escalation ladder + `display()`-as-separate-call discipline** (aside-07 patterns 1-2) — Silver already has `screenshot`/`snapshot` as distinct verbs (F5); the missing piece is a *documented* escalation order in the skill guide (text-snapshot before screenshot), which is a `SKILL.md` edit, not new code, in either base.

### Important, adopt with adaptation
6. **`memory_search`-style markdown-first memory with grep fallback** (aside-06 patterns 1, 5, 7) — keyless-compatible IF embeddings are treated as optional (per aside-06's own core insight: markdown is truth, vectors are a disposable cache). Neither base has any memory directory today (F6), so this is greenfield in both. TS is marginally easier (npm ecosystem has more turnkey local-embedding options: `@xenova/transformers`, ONNX bindings) but a **grep-only** memory tier (no embeddings at all) is equally trivial in Rust (`ripgrep`-style search over `memory/*.md`) and fully keyless either way — recommend shipping grep-only first, embeddings as a later opt-in layer, independent of language choice.
7. **Subagent-as-child-session with a restricted tool allowlist** (aside-06 patterns 10-12) — Silver's daemon-per-session model (F1) and agent-browser's sidecar-per-session model (F2) both already have "session" as a first-class addressable unit, which is exactly the substrate Aside's subagent design reuses ("spawn a subagent = create another session, restricted tool-wise, track its id" — aside-02 pattern 18). This is *more* natural on Silver's Rust daemon (F1) since daemon-tracked sessions already have process-level identity (`walk_daemons`, `is_pid_alive` per `main.rs:33-34`) to hang a `parent_session_id`/concurrency-cap/depth-guard off of; agent-browser's sidecar-JSON model (F2) can do the same via an added `parentSession`/`isSubagent` field in the sidecar file, slightly more plumbing since there's no central daemon registry to enforce a concurrency cap against (would need a lockfile/semaphore, per aside-06 anti-pattern 8's own recommended CLI-shape fallback).
8. **Skill auto-injection matcher (URL-glob specificity score)** (aside-07 patterns 14-16) — additive to the existing static `SKILL.md` model (F7): Silver already ships per-plugin `SKILL.md` files (`silver/skill-data/*/SKILL.md`), so layering a URL/keyword-scored "auto-attach the right one" resolver on top is a retrieval-layer addition, not a rewrite, in either language (string-glob-matching is trivial in both Rust `glob`/`regex` crates and TS `minimatch`).

### Nice / defer
9. **Sandboxed library-attach-on-navigate REPL model with `node:vm`** (aside-07 patterns 17-18) and **the full persistent multi-statement `repl` code-exec tool** (aside-02 pattern 1) — this is the pattern where language choice matters most (F8, F3). TS/agent-browser can approximate it relatively cheaply via `node:vm` with injected Playwright globals (`page`, `openTab`, `fs`) — the same primitive Aside itself uses (F8, aside-07 pattern 18). Silver/Rust has no equivalent std primitive; embedding a JS engine (`rquickjs`/`boa`) purely to offer a persistent scripting surface is a heavyweight dependency addition that partially duplicates what the *host* LLM's own coding/bash tools already provide when Silver runs as a skill inside an agent harness. Recommend: **do not build a bespoke persistent REPL tool in Silver's basement at all** — Silver's `eval` verb already gives one-shot page-JS execution (F3), and a host agent (Claude Code, etc.) already has its own persistent shell/Bash tool it can loop `agent-browser eval ...` through, which achieves the same "compose control flow across actions" benefit Aside's `repl` provides, without Silver needing to embed an interpreter. This directly reflects aside-06's own anti-pattern 4 guidance (don't port daemon-scale machinery into a CLI-invoked-per-task tool) generalized to the REPL pattern.
10. **Gated "dreaming" memory consolidation, content-addressed history/revert log** (aside-06 patterns 8-9, aside-06 anti-pattern 1) — aside-06 itself flags this as overkill for a stateless/per-task CLI skill; skip for v1 in both bases. Revisit only if Silver grows a long-lived persistent-agent deployment mode.
11. **Command envelope (`steer`/`queue`/`interrupt`) and resumable `runId`/`seq` event stream** (aside-02 patterns 5-7) — valuable only if Silver builds its own long-running agent-loop daemon (not just a browser-session daemon, which is all F1 currently is). Silver's daemon (F1) is the closer starting substrate for this than agent-browser's sidecar model (F2) if ever pursued, but neither base currently runs an agent loop at all — Silver is "eyes+hands," never a model caller, per the project brief — so this pattern has no home unless the project scope changes.

## Relevance to the 9 criteria (compact)

- **(a) agent-ergonomic CLI + SKILL.md:** citation-ID + escalation-ladder + auto-injection matcher (items 1, 5, 8) all improve this without new architecture.
- **(b) fast quick tasks:** unaffected/improved — none of the recommended items add per-call latency; deferred REPL avoids adding interpreter startup cost.
- **(c) long-running/resumable tasks:** memory (item 6) and subagent-as-session (item 7) are the direct contributors; command-envelope/event-stream (item 11) deferred until an agent-loop daemon exists.
- **(d) parallel multi-agent orchestration:** item 7 is the load-bearing recommendation; both bases' existing per-session daemon/sidecar identity (F1, F2) is the right substrate, Rust daemon slightly ahead on enforcement primitives.
- **(e) install-and-use, zero config:** grep-only memory tier and deferred REPL keep the zero-config bar; embeddings/vector index framed as strictly optional (per aside-06's own core insight).
- **(f) keyless:** every recommended item (1,2,3,4,5,6,7,8) is keyless-compatible as specified — none requires Silver itself to call a model; OCR/vision-model-routing patterns (aside-07 patterns 8-9, 12) are explicitly OUT because they require Silver to own a model call, violating the "Silver never calls a model" invariant, and are excluded from all recommendations above.
- **(g)-(i) [security/consolidation/etc., implicit]:** untrusted-output wrapping (item 3) and the "don't scatter tool-gating" refactor (item 2) are the direct security/quality contributions; both are additive to existing `security/` modules already present in agent-browser (F4) and Silver's confirm/validation modules.
